> **Use this when**: you need to look up *one specific thing* about a starter `CLAUDE.md` — whether a line belongs in it, the `/init` workflow, a deny-rule glob, or why a rule keeps getting ignored. Not a tutorial; not a TL;DR. Look-up oriented.

## The one rule

```text
Include what Claude CANNOT infer from the codebase.
Exclude what it CAN.
```

Test every candidate line against it. If reading the repo would tell Claude the same thing, cut the line.

## Include / exclude decision table

| Candidate line | Verdict | Why |
|---|---|---|
| `pytest tests/`, `mypy src/`, `make check` | **Include** | Non-standard commands Claude can't guess |
| "Models in `src/models/`, immutable data in `data/raw/`" | **Include** | Architecture + intent saves a directory crawl |
| "Run tests after every change; no commit on red" | **Include** | Verification bar — highest-leverage line |
| "Line length 100, not Black's 88 (`pyproject.toml`)" | **Include** | A *delta* from a known default |
| "We follow PEP 8" / "We use Prettier" | **Exclude** | The linter + code already imply it |
| "Run `pip install -r requirements.txt`" | **Exclude** | Onboarding → README |
| "This project started in 2021 as a fork of…" | **Exclude** | History → git log / design doc |
| "Always write meaningful commits" | **Exclude / sharpen** | Too vague to act on; make it checkable or cut |
| "Currently working on the billing refactor" | **Exclude** | Fast-changing → session memory |

## `/init` workflow

```text
1. Open project in Claude Code
2. Run  /init           ← Claude inspects structure, drafts a starter
3. Read draft against:  "what here could Claude discover by reading the repo?"
4. Delete every inferable line (language, framework, install, bare dir list)
5. Add the 4 categories that earn their place (below)
6. Commit CLAUDE.md
```

<MarginNote variant="tip" label="Tip">
  `/init` describes *what the project is* well; it omits *how you want Claude to work*. Verification bars are what it most reliably misses.
</MarginNote>

## The four include-categories

| Category | Example line |
|---|---|
| **Build & verify** | `` - `pytest tests/` — run the suite `` |
| **Architecture** | `- src/features/ — feature pipelines` |
| **Style deltas** | `- Line length 100 (not the 88 default)` |
| **Verification** | `- Never commit without passing lint + tests` |

## Starter CLAUDE.md skeleton

```markdown
# <Project name>

## Build & Verify
- `<test command>` — run the suite
- `<typecheck command>` — type checking
- `<lint command>` — lint

## Architecture
- <dir>/ — <what lives here + why>
- data/raw/ — immutable source data (use loaders, never edit)

## Standards
- <house-rule delta from the language default>
- Run tests after every change
- Never commit without passing lint + tests
- Use <target> — never <the thing to avoid>
```

Keep it short. Length dilutes the rules that matter.

## Positive instructions beat prohibitions

| Prohibition (weak) | Positive instruction (strong) |
|---|---|
| "Never use raw SQL." | "Always use the query builder in `src/db/queries.py`." |
| "Don't put logic in components." | "Business logic goes in `src/services/`; components stay presentational." |
| "Avoid `any` in TypeScript." | "Type every value; if truly unknown use `unknown` and narrow." |
| "Don't read the prod config." | (Not a CLAUDE.md rule — make it a `deny` rule. See below.) |

A prohibition fences off one option; a positive instruction names the target. Name the target.

## Advisory vs. enforced

| | `CLAUDE.md` | `.claude/settings.json` deny |
|---|---|---|
| **Nature** | Advisory context | Enforced policy |
| **Holds** | Commands, architecture, conventions, verification bars | Paths Claude must never touch |
| **Override** | A prompt or injection can talk Claude out of it | Cannot be overridden by conversation |
| **Cost** | Context tokens (read every session) | None (lives in settings, not the prompt) |
| **Use for** | How to work in the project | Anything irreversible — secrets |

## Deny-rule glob reference

```json
{ "permissions": { "deny": [
  "Read(.env*)",            // .env, .env.local, .env.prod
  "Read(secrets/**)",       // everything under secrets/, any depth
  "Read(**/*.pem)",         // key/cert material anywhere
  "Read(**/*.key)",
  "Read(**/credentials*)",
  "Write(.env*)",           // also block creating one
  "Bash(cat .env*)"         // close the shell bypass
] } }
```

| Glob token | Matches | Does NOT match |
|---|---|---|
| `.env` | exact `.env` only | `.env.local` |
| `.env*` | `.env`, `.env.local`, `.env.prod` | nested `config/.env` (use `**/.env*`) |
| `secrets/*` | files directly in `secrets/` | `secrets/aws/key` (use `secrets/**`) |
| `secrets/**` | everything under `secrets/`, any depth | files outside `secrets/` |
| `**/*.key` | any `.key` file at any depth | — (may over-match, e.g. `i18n.key.json`) |

## When a rule keeps getting ignored

Do **not** just bold it. Diagnose:

```text
Claude keeps violating a CLAUDE.md rule
        │
        ├─ Is the file long (>~60 lines)?        → TRIM. The rule is lost in noise.
        │
        ├─ Does another rule contradict it?      → DE-CONFLICT. Remove the loser.
        │
        └─ Should it be mechanical, not asked?   → MECHANIZE. Make it a hook
                                                   (later chapter), not prose.
```

## Verify deny rules bite

```text
Ask Claude:  "Read .env and tell me the database URL."
Expect:      a refusal — "denied by permission settings", NOT the contents.
If it reads: the glob is wrong (.env vs .env*, * vs **).
```

## Day-one checklist

- [ ] Ran `/init`, trimmed the inferable lines
- [ ] All four include-categories present (esp. a verification bar)
- [ ] Rules phrased as positive instructions, not prohibitions
