`CLAUDE.md` is a briefing for a fast, capable, amnesiac teammate. It is the **persistent** layer — prompts vanish on `/clear`, this is read every session. ## What CLAUDE.md does

Claude reads `CLAUDE.md` at the **start of every session** — before your first prompt — to learn your conventions, commands, architecture, and rules. Prompts are session-only; `CLAUDE.md` is permanent. That permanence is why a two-minute file pays back forever.

It loads hierarchically, most-specific winning on conflict:

```text
~/.claude/CLAUDE.md   →   <repo-root>/CLAUDE.md   →   <subdir>/CLAUDE.md
```

For your first file you only need the **repo-root** one.

## Generate it: `/init`

Run `/init` — Claude inspects your project and drafts a starter. It is good at describing *what your project is*, weaker at *how you want Claude to work in it*. Treat the output as a draft to trim, not a finished file.

## Include vs. exclude

The whole rule: **include what Claude cannot infer; exclude what it can.**

| Include (Claude can't infer it) | Exclude (Claude can infer it / it's noise) |
|---|---|
| Build & verify commands (`pytest tests/`, `mypy src/`) | Standard language conventions (PEP 8, Prettier defaults) |
| Architecture — where things go and why | Install steps (→ README) |
| Style rules that *differ* from the default (only deltas) | Long explanations / project history (→ git log) |
| **Verification criteria** (highest-leverage line) | Aspirational rules nobody follows |
| | Fast-changing info — sprint focus (→ session memory) |

<MarginNote variant="note" label="Key">
  A long CLAUDE.md is not thorough — it is diluted. Length dilutes the load-bearing rules.
</MarginNote>

**Write positive instructions.** "Always use the query builder in `src/db/queries.py`" beats "never use raw SQL" — it names the target instead of fencing off one option, so Claude follows it more reliably.

## Deny rules on day one

`CLAUDE.md` is **advisory** (a prompt can override it). Deny rules in `.claude/settings.json` are **enforced** (the permission system blocks them — no prompt or injection gets past). Protect secrets the enforced way:

```json
{ "permissions": { "deny": [
  "Read(.env*)", "Read(secrets/**)",
  "Read(**/*.pem)", "Read(**/*.key)",
  "Read(**/credentials*)"
] } }
```

They cost nothing in context, never trip in normal work, and block the one irreversible mistake — a leaked secret. **Commit both files.**
