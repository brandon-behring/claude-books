## The layer pyramid

Cheaper at the bottom, more expensive at the top. Layer 7 (domain correctness) sits *above* the automated stack — human-only.

## Layer reference

| # | Layer | Catches | Tooling examples |
|---|---|---|---|
| 1 | Type check | Shape errors, null bugs, refactor drift | `mypy`, `pyright`, `ty`, `tsc --noEmit`, compiler |
| 2 | Lint | Style, dead code, common bug patterns | `ruff`, `eslint`, `clippy`, `golangci-lint` |
| 3 | Unit | Logic bugs; edge cases | `pytest`, `vitest`, `cargo test`, `go test` |
| 4 | Integration | Module-boundary bugs (DB / HTTP / config) | `pytest -m integration`, testcontainers, `msw` |
| 5 | E2E / smoke | Full-stack regressions | Playwright, Cypress, `curl` smoke scripts |
| 6 | Manual | UX, visual, copy, a11y | Human + browser + screenreader |
| 7 | Domain correctness | "Solves the right problem" | Human + spec + worked example |

## Phase × layer matrix

`R` = required; `r` = recommended; `—` = defer / skip.

| Layer | Prototype | MVP / internal | Production | Critical-path |
|---|---|---|---|---|
| 1 — type check | **R** | **R** | **R** | **R** |
| 2 — lint | **R** | **R** | **R** | **R** |
| 3 — unit | **R** (smoke) | **R** | **R** | **R** (property-based) |
| 4 — integration | — | **R** | **R** | **R** |
| 5 — e2e / smoke | — | r | **R** | **R** |
| 6 — manual | — | r per change | **R** on user-facing | **R** (formal review) |
| 7 — domain correctness | r | **R** per change | **R** per change | **R** + audit log |

## CLAUDE.md verification-criteria templates

### Prototype phase

```markdown
## Verification (prototype phase)

Before claiming done:
1. `mypy src/` — must pass
2. `ruff check src/` — must pass
3. Sketch the happy path in a one-line example and
   include it in the response.

Tests + integration deferred until concept validated.
```

### MVP / internal-tool phase

```markdown
## Verification (MVP phase)

Before claiming done:
1. `mypy src/`
2. `ruff check src/ tests/`
3. `pytest tests/ -x --quiet`
4. For numerical / statistical code: include one worked
   example with a known answer.

Never claim "done" if 1, 2, or 3 fails.
```

### Production phase

```markdown
## Verification (production phase)

Before claiming done:
1-4 as MVP, plus:
5. `pytest tests/integration/`
6. For UI changes: smoke the happy path in a real browser
   and paste the URL / screenshot path.

For non-UI changes, run the relevant e2e suite via:
   `npm run e2e -- --grep "<area>"`

If you cannot run a check, say so explicitly. Never assume.
```

## Hook event reference (verification gates)

| Event | Fires when | Use to block | Caveat |
|---|---|---|---|
| `PreToolUse` | Before any tool call | Destructive commands; out-of-policy paths | Runs on every tool call — keep cheap |
| `PostToolUse` | After a tool call returns | Sanity-check the edit (e.g., re-typecheck the file) | Add file-scope matcher to avoid global cost |
| `Stop` | Before turn-end ("done") | The verification gate (mypy + lint + unit) | Most natural place for the "did you actually verify?" check |
| `PreCompact` | Before `/compact` | Capture state before context loss | Useful for long sessions; not a verification gate per se |
| `SessionStart` | Beginning of a session | Print "current phase = MVP" reminder | Use sparingly; spam dilutes signal |

Non-zero exit blocks the triggering action. Stderr is surfaced to Claude as feedback.

## Minimal `Stop` hook (copy-paste)

`.claude/settings.json`:

```json
{
  "hooks": {
    "Stop": [
      {
        "matcher": ".*",
        "hooks": [
          {
            "type": "command",
            "command": "bash .claude/scripts/verify.sh"
          }
        ]
      }
    ]
  }
}
```

`.claude/scripts/verify.sh`:

```bash
#!/usr/bin/env bash
set -euo pipefail
mypy src/ || { echo "FAIL: mypy" >&2; exit 1; }
ruff check src/ tests/ || { echo "FAIL: ruff" >&2; exit 1; }
pytest tests/ -x --quiet || { echo "FAIL: pytest" >&2; exit 1; }
echo "OK: gate passed"
```

`chmod +x .claude/scripts/verify.sh`.

## Test-style decision aid

| Symptom | Reach for | Why |
|---|---|---|
| "I keep getting off-by-one bugs in this function" | Property-based test (Hypothesis / fast-check) | Generators surface inputs you didn't think of |
| "Tests pass, prod breaks" | Add layer 4 (integration) | Module boundary bug, not logic bug |
| "Everything passes, user reports wrong number" | Layer 7 — run on real data, eyeball | Domain correctness gap |
| "Refactor broke unrelated thing" | Type check (layer 1) + unit (layer 3) | Type check should have caught signature drift |
| "Flaky CI" | Move to lower layer if possible; isolate the flake; never `@retry` it away | Flakes hide real bugs |
| "Slow test suite, devs disable it" | Scope the hook (`pytest -k`, `--lf`); full run in CI | Slow gate gets bypassed; bypassed gate is no gate |

## Session-start checklist (verification edition)

- [ ] What phase is this project in? (Written down somewhere?)
- [ ] Which layers are required for this phase?
- [ ] Is the verification criteria block in `CLAUDE.md` current?
- [ ] Is there a hook enforcing the non-negotiable layers?
- [ ] When the hook fires, can Claude actually read the failure message? (Stderr, not stdout.)
- [ ] For numerical / data work: is there a "known-answer" example you can spot-check?
