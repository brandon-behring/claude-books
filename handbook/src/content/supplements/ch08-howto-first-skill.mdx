> **You have**: a release procedure that you currently re-type into Claude every release (or worse, forget half of). Things like "bump the version, regenerate the changelog from conventional-commits, run the full test matrix, tag the commit, post to the release channel." **You want**: a `.claude/skills/release-checklist/SKILL.md` that any session in this repo can invoke as `/release-checklist`, with Claude executing the procedure consistently every time.

## Prerequisites

- Claude Code installed and configured for the repo (CLAUDE.md present).
- The repo has a real release procedure (even an informal one in a wiki or in your head).
- 10 minutes.

## Steps

### 1. Decide that a skill is actually the right layer

Walk the decision framework from the tutorial:

- **Is this a reusable prompt?** It's reusable, but it's not just a prompt — it's a multi-step procedure with branches (test failures, version bump rules, etc.).
- **Is this a reusable procedure?** Yes. → **Skill.**

If your "procedure" is really one paragraph of instructions with no branches, stop here and make a slash command instead (`.claude/commands/release.md`). Skills are for the cases where there's enough structure that a command would feel cramped.

### 2. Create the skill file

In the repo root:

```bash
mkdir -p .claude/skills/release-checklist
touch .claude/skills/release-checklist/SKILL.md
```

<MarginNote variant="note" label="Cross-Ref">
  Project-scoped skills live at <code>.claude/skills/&lt;name&gt;/SKILL.md</code>; user-scoped (available across all your repos) at <code>~/.claude/skills/&lt;name&gt;/SKILL.md</code>. Same directory format.
</MarginNote>

### 3. Write the frontmatter

Skills use YAML frontmatter to declare the skill's identity and (optionally) its allowed tools.

```markdown
---
name: release-checklist
description: Run our standard release procedure end-to-end — version bump, changelog regen, full test matrix, tag, and release channel post.
---
```

The `name` field is what the skill is invoked as (`/release-checklist`). The `description` is what Claude reads to decide whether the skill applies to the current request — make it specific enough that "release" is unambiguous (versus, say, a "release notes" skill).

### 4. Write the procedure body

The body is the procedure itself, written as if you were explaining it to a careful, methodical new hire. Be specific. Be explicit about checkpoints where Claude should pause and report rather than power through.

```markdown
# Release checklist

## Inputs
- Release version (semver, e.g. `1.4.0`). If not provided, infer from
  conventional commits since the last tag (patch / minor / major).
- Optional: release-channel target (defaults to `#releases`).

## Phase 1 — Pre-flight
1. Verify the working tree is clean (`git status`). If not, STOP and report.
2. Verify the current branch is `main`. If not, STOP and report.
3. Pull latest (`git pull --ff-only`). If non-FF, STOP and report.
4. Run the full test suite (`pnpm test`). If anything fails, STOP and
   report the failing tests — do not proceed.

## Phase 2 — Version bump + changelog
5. Update `package.json` `version` to the target version.
6. Regenerate `CHANGELOG.md` from conventional commits since the last tag,
   grouped by `feat:` / `fix:` / `chore:` / `docs:`.
7. Show me the diff and WAIT for my confirmation before proceeding.

## Phase 3 — Tag + push
8. Commit with message `chore: release v<version>`.
9. Tag with `v<version>`.
10. Push the commit AND the tag (`git push && git push --tags`).

## Phase 4 — Announce
11. Post to the release-channel target with: version, top 3 changes,
    link to the CHANGELOG diff.
12. Report the release URL.

## Things to remember
- Never skip Phase 1 even if I'm in a hurry.
- Never auto-confirm Phase 2 step 7 — always show the diff and wait.
- If any test is flaky, report it but do not retry — flakiness is its
  own bug.
```

<MarginNote variant="tip" label="Tip">
  The "STOP and report" markers are load-bearing. Without them, Claude is biased to power through transient failures. With them, you get a halt-on-error procedure by construction.
</MarginNote>

### 5. Invoke the skill

In a fresh Claude Code session in the repo:

```
/release-checklist 1.4.0
```

Or, in natural language:

```
Run our release checklist for v1.4.0.
```

Claude finds the skill (via the `description` match), loads the body, and executes it phase by phase. You'll see the diff at step 7 and be asked to confirm.

### 6. Iterate on the first real failure

The first time you use the skill on a real release, you'll find a gap — a step that was implicit in your head but missing from the file. Add it. Skills are living procedures; the version-controlled file lets the whole team benefit when one person discovers the gap.

<KeyIdea>
A skill captures a procedure once so every future session executes it consistently. The version-controlled file in <code>.claude/skills/</code> turns your individual checklist into the team's checklist. The first release after writing it is the one that catches the gaps; every release after that is the payoff.
</KeyIdea>

### 7. (Optional) Promote to user-scoped if it's universal

If this same checklist applies across all your repos with minor parameterization, move it to `~/.claude/skills/release-checklist/SKILL.md`. Per-repo CLAUDE.md files can still override the specifics (channel name, test command, etc.).

## Expected outcome

You have `.claude/skills/release-checklist/SKILL.md` committed to the repo. Any session — yours or a teammate's — that says "run our release checklist" gets the same procedure, executed consistently, with explicit halt-on-error checkpoints. Future-you (and future hires) inherit the procedure without you having to re-explain it.

## When this fails

- **Claude ignores the skill.** Usually the `description` is too generic ("releases") and doesn't match the request. Sharpen it: "Run our standard release procedure end-to-end — version bump, changelog regen, full test matrix, tag, push, channel post." The description is Claude's matching signal.
- **Claude powers through a failing test.** You forgot a "STOP and report" marker, or the marker is too soft ("note this and continue"). Tighten the wording to "STOP and report. Do not proceed."
- **The skill drifts from reality.** The team changed the deploy procedure but didn't update the skill. Treat skills like code — they belong in code review when the procedure changes.
