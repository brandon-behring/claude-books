## Shared CLAUDE.md in version control

The whole team standard lives in three files, and the split between them is the single most important governance decision you make. The project-level `CLAUDE.md` (structured per Chapter 2 _(forthcoming)_) and `.claude/settings.json` are **committed to version control**, so every team member gets the same project context, the same permissions, and the same hooks. Claude Code is available on the Team and Enterprise plans, where a shared `CLAUDE.md` in the repo is the intended way to distribute that context across seats.

Three files form the team standard:

- **`CLAUDE.md`** — project knowledge: build commands, architecture, conventions. Reviewed in PRs like any other code.
- **`.claude/settings.json`** — team permissions: deny rules, required hooks. Committed to git.
- **`.claude/settings.local.json`** — personal preferences: model choice, output style, personal API tokens. Auto-gitignored.

<InsightBox title="Why separate committed and local settings">
  Put model choice in the committed `settings.json` and every developer who needs a more capable model for a hard task must override the team file to get it — and their override drifts from the shared standard. Put it in `settings.local.json` and each developer chooses their own model without touching anyone else's setup. The principle is a clean line: **team standards** (deny rules, required hooks) are committed; **personal preferences** (model, output style) are local.
</InsightBox>

<Tip n="31" title="Commit team standards; gitignore personal preferences.">
`CLAUDE.md` and `settings.json` carry the conventions, deny rules, and hooks the whole team shares; `settings.local.json` carries each person's model and style. Drawing that line keeps one developer's preference from silently becoming everyone's policy.
</Tip>

## Managed settings for IT

A committed `settings.json` is a team _convention_ — any developer can still edit it, and a personal `settings.local.json` can relax it. When an organization needs policy that developers **cannot** override, that is a different layer. IT administrators can deploy `managed-settings.json` files that override all other settings.<Citation src="anthropic-claude-code-settings" /> This enables:

- Organization-wide deny rules (sensitive-file access)
- Mandatory hooks (security scanning, compliance checks)
- Approved MCP-server allowlists
- Model restrictions (e.g., a cheaper tier only, for cost control)

Managed settings deploy to system-level paths and **cannot be overridden** by project or user settings. They sit at the top of the five-level precedence hierarchy — managed/policy settings outrank command-line args, which outrank local, then project, then user settings (Chapter 9 _(forthcoming)_ has the full hierarchy). Managed settings are the IT equivalent of a required hook: non-negotiable.

<KeyIdea>
There are two kinds of "shared" configuration, and conflating them is the root of most team-governance confusion. **Committed `settings.json`** is a convention the team agrees to and any teammate can edit in a PR. **Managed `managed-settings.json`** is policy that IT deploys to system paths, that no project or user setting can override, and that a developer cannot edit away. Use committed settings for what the team chooses together; use managed settings for what the organization must enforce.
</KeyIdea>

### Delivery mechanisms

Managed settings can be delivered through several channels, all at the highest priority level.<Citation src="anthropic-claude-code-settings" /> The mechanism you pick is a function of how your fleet is managed:

- **Server-managed** — delivered from Anthropic servers via the admin console (Teams/Enterprise). Fetched at startup and polled periodically, so policy updates reach seats without touching each machine.
- **macOS MDM** — a managed-preferences domain (`com.anthropic.claudecode`) pushed through tools like Jamf or Kandji.
- **Windows Registry** — a policy key under `HKLM\SOFTWARE\Policies\ClaudeCode` carrying the settings payload, deployed via Group Policy or Intune.
- **File-based** — a `managed-settings.json` at a system path (for example `/etc/claude-code/` on Linux, or the system Application Support directory on macOS).

A drop-in directory (`managed-settings.d/`) alongside `managed-settings.json` accepts policy **fragments** that merge alphabetically — use numeric prefixes (`10-security.json`, `20-models.json`) to make the merge order explicit. Server-managed delivery requires a Teams or Enterprise plan and a recent Claude Code build; the surface evolves, so confirm the current minimum version when you roll it out.

<MarginNote variant="warning" label="Warning">
  Server-managed settings are a client-side control. For unmanaged or BYO devices where you cannot trust the client, reach for MDM-level enforcement — it is harder to circumvent than a setting the client itself applies.
</MarginNote>

### Managed-only settings

A handful of keys are enforced **only** at the managed level — they are the mechanisms that turn "we have a policy" into "the policy is enforced" (set in a project or user file, a developer could just relax them; in `managed-settings.json` they cannot):

- **`allowManagedPermissionRulesOnly`** — only managed allow/deny rules apply; users and projects cannot add their own permission rules on top.
- **`availableModels`** — restrict which models users can select (the main session, subagents, and fallback chains all honor it).
- **`companyAnnouncements`** — display messages to users at startup (policy reminders, links to AI-usage guidelines).
- **`disableBypassPermissionsMode`** — prevent the permission-skipping mode, so no one can opt out of the permission system entirely.

A minimal `managed-settings.json` that pins the permission surface, the model allowlist, and a startup notice:

```json
{
  "permissions": {
    "deny": ["Bash(curl *)", "Read(./.env)"],
    "disableBypassPermissionsMode": "disable"
  },
  "availableModels": ["sonnet", "haiku"],
  "companyAnnouncements": [
    "Review AI usage guidelines at docs.acme.com/ai"
  ]
}
```

<Tip n="32" title="Use managed settings only for what must be enforced.">
Anything a developer should be able to relax for a good reason belongs in committed `settings.json`. Reserve `managed-settings.json` — and the managed-only keys like `allowManagedPermissionRulesOnly` and `availableModels` — for the deny rules, hooks, and model limits the organization genuinely cannot allow a teammate to override.
</Tip>

When hooks are configured at multiple levels they **all run** — managed, project, and local hooks fire for the same event. If they conflict (a managed hook enforces formatting that a local hook undoes), the managed hook wins. To debug unexpected hook behavior, view the merged configuration with `/hooks`, check each level's `settings.json`, and remember that a non-zero hook exit code blocks the action that triggered it — test a hook in isolation before promoting it into managed settings.

## Onboarding new team members

The payoff of committed configuration is onboarding speed. Day one with Claude for a new teammate is a short, repeatable sequence:

1. **Clone the repo.** `CLAUDE.md` and `settings.json` load automatically — context and permissions arrive with the checkout.
2. **Run `/init`** to verify Claude understands the project, and compare its output to the existing `CLAUDE.md`; they should align. A divergence is a signal that the committed file has drifted from the code.
3. **Create `.claude/settings.local.json`** for personal preferences (model, output style).
4. **Run the team's `/check` command** to verify the environment works end to end.
5. **Review the team's hook configuration** — what runs on file write? On commit? Why? Understanding the guardrails is part of onboarding, not an afterthought.

<KeyIdea>
The target is that a new developer is productive with Claude **within their first day, without reading a setup guide**. That only happens if the configuration in version control is self-documenting — the `CLAUDE.md`, the deny rules, and the hooks have to explain the project well enough that cloning the repo _is_ the onboarding. If onboarding still needs a separate wiki page, that page's content belongs in the committed files instead.
</KeyIdea>

## Shared extensions at scale

Chapter 9 _(forthcoming)_ introduced the extension hierarchy — commands, skills, hooks, and MCP servers. At team scale each of these needs governance, not just authorship:

- **Shared skills** — commit team skills to `.claude/skills/` in the repo so quality dashboards, pre-release checklists, and onboarding workflows benefit everyone. For an expensive skill that does a lot of reading, set `context: fork` so it runs in its own context window and does not spend teammates' context budgets on its intermediate work.
- **Managed MCP** — deploy IT-approved MCP servers through managed settings. An allowlist prevents ad-hoc server installation while still giving the team standardized access to shared databases, ticketing systems, and monitoring tools.
- **Shared hook libraries** — keep common hooks (auto-format, test gates, lint) in a central repository and import them via a hub-and-spoke pattern (a shared base that projects reference, covered in Chapter 9 _(forthcoming)_). Update the hook once and every project inherits the change.

The hierarchy scales naturally along the committed-versus-managed line from earlier in this chapter: commands and skills live **in the repo** for the team to share and edit, hooks live in **`settings.json`** for enforcement the team agrees to, and managed MCP lives in **managed settings** for the integrations IT controls.

## Code-review patterns with Claude

Two patterns put Claude into a team's review loop. Both follow one rule: Claude does the analysis, a human makes the decision.

### Pre-PR self-review

Before opening a PR, have Claude review your own changes so the obvious problems never reach a reviewer:

```text
Review the diff in `git diff main...HEAD` for:
1. Logic errors and bugs
2. Missing test coverage
3. Style violations per our CLAUDE.md
4. Security concerns
Report issues with file:line references and severity.
```

This catches the easy issues before reviewers see the code, which shortens review cycles and lets human attention go to design rather than typos.

### Reviewer-assist

During review, point Claude at a specific concern to investigate the blast radius of a change:

```text
In PR #234, the author changed the feature normalization
strategy in src/features/preprocessing.py. What are the
implications for models trained on the old normalization?
Check whether any downstream code depends on the old
feature distributions.
```

<Tip n="33" title="AI augments review; it does not replace it.">
In both patterns the reviewer still owns the decision — Claude supplies analysis, file:line references, and a read of the downstream impact. Treat its output as a well-prepared briefing for a human judgment call, not the judgment itself.
</Tip>

## Common pitfalls

<Pitfall title="Putting personal preferences in the committed settings.json">
Model choice, output style, and personal tokens belong in `settings.local.json` (auto-gitignored). Commit them and one person's preference becomes everyone's policy — and teammates start overriding the shared file, which is exactly the drift committed settings are supposed to prevent.
</Pitfall>

<Pitfall title="Using managed settings for team conventions">
Managed settings are policy a developer cannot override. Use them for what the organization must enforce (deny rules on secrets, mandatory security hooks, a model allowlist) — not for ordinary team conventions that a teammate might reasonably need to relax. Over-locking managed settings turns every legitimate exception into an IT ticket.
</Pitfall>

<Pitfall title="Relying on a committed deny rule for security">
A deny rule in a committed `settings.json` is a convention: a developer can edit the file or add a `settings.local.json` that loosens it. If a rule is a real security boundary, it has to live in managed settings (with `allowManagedPermissionRulesOnly`), where no user or project setting can relax it.
</Pitfall>

<Pitfall title="Onboarding that needs a separate setup guide">
If a new teammate can't become productive from the cloned repo alone, the committed configuration isn't self-documenting yet. The fix is not a longer wiki page — it's moving that page's content into `CLAUDE.md`, the deny rules, and the hooks, so the checkout carries everything.
</Pitfall>

<Pitfall title="Treating Claude's review as the decision">
Pre-PR self-review and reviewer-assist surface issues and trace downstream impact, but a human still owns the merge call. Auto-merging on a clean Claude review removes the judgment the review was meant to inform.
</Pitfall>

## Quick reference

- Commit **`CLAUDE.md`** and **`.claude/settings.json`** for team standards; keep **`.claude/settings.local.json`** for personal preferences (auto-gitignored).
- **Managed settings** (`managed-settings.json`) enforce IT policy at the highest precedence — they cannot be overridden by project or user settings.
- **Managed-enforced keys**: `allowManagedPermissionRulesOnly` (lock the permission rules), `availableModels` (restrict selectable models), `companyAnnouncements` (startup messages), `disableBypassPermissionsMode` (block permission-skipping).
- **Delivery**: server-managed (admin console), macOS MDM, Windows Registry, or file-based; fragments in `managed-settings.d/` merge alphabetically (numeric prefixes set order).
- **Hooks at multiple levels all run**; on conflict, managed wins. Debug with `/hooks` and remember a non-zero exit blocks the action.
- New teammates are productive **day one** from the committed configuration — `clone → /init (compare) → settings.local.json → /check → review hooks`.
- **Code review**: pre-PR self-review catches issues before humans see them; reviewer-assist investigates a specific concern. The human always decides.

<Solution for="ch15-ex-1">
Model choice is a **personal preference**, so it belongs in `settings.local.json` (which is auto-gitignored), not in the committed `settings.json`. By committing it, the teammate made their personal model the team default — and because settings merge with committed values overriding user-level ones, everyone else's model now "switches" to the committed choice whenever the project settings apply. The fix is to revert the committed change and have each developer set their own model in `settings.local.json`, keeping the committed file to deny rules and required hooks.
</Solution>

<Solution for="ch15-ex-2">
`allowManagedPermissionRulesOnly` guarantees developers cannot add their own permission rules (only managed allow/deny rules apply), and `disableBypassPermissionsMode` guarantees they cannot turn the permission system off (it blocks the permission-skipping mode). Putting the same deny rules in a committed `settings.json` doesn't deliver either guarantee because committed settings are a convention, not policy: a developer can edit the committed file in a PR, or add a `settings.local.json` that loosens the rules, and nothing stops them from running in a mode that bypasses permissions. Only the managed layer — which project and user settings cannot override — makes the boundary actually binding.
</Solution>

<Solution for="ch15-ex-3">
Trust the **committed `CLAUDE.md`** as the team's agreed standard, but treat the disagreement as a bug report. `/init` proposes conventions from what the code _currently looks like_; the committed file states what the team has _decided_. When they diverge, either the code has drifted from the documented conventions (some files no longer follow the standard) or the committed file is stale (it documents a convention the team has since abandoned). Either way, the disagreement is a prompt to reconcile the two — update the code to match the doc, or update the doc to match reality — so the next new teammate sees them aligned.
</Solution>
