## What Claude Code actually is

Claude Code is an agentic coding tool that operates directly in your terminal.<Citation src="claude-code-overview" /> It runs an **agent loop**: it reads your prompt, reasons about the task, selects a tool (read a file, edit code, run a command), observes the result, and decides what to do next.<Citation src="anthropic-building-effective-agents" /> The cycle repeats until the task is complete or Claude stops to ask you something.

<Figure

  caption="The agent loop: reason → decide → call → observe → repeat. Compiled from TikZ source at figures/agent-loop/agent-loop.tex via the build-figures pipeline."
  width="80%"
/>

Three properties define the system. Every technique in this book is, at bottom, a way of working _with_ one of them rather than against it.

- **Context window** — Everything Claude knows about your task (your prompt, files it has read, tool results, conversation history) occupies a finite window. When the window fills, older information fades. Context is not infinite memory; it is a workspace.
- **Tool use** — Claude does not type code into a file. It calls tools: `Read`, `Edit`, `Write`, `Bash`, `Glob`, `Grep`. Each tool call consumes context (both the request and the result) and produces observable effects. Every call has a cost in tokens.
- **Configuration layers** — Claude's behavior is shaped by four layers: `CLAUDE.md` (project knowledge), rules (conditional context), settings (permissions and behavior), and output styles (communication patterns). These are not suggestions — they are the operating system through which Claude interprets your project.

<InsightBox title="Why the mental model matters">
  Developers who treat Claude as a chatbot prompt it like one: vague requests, no verification, no context management. Developers who understand the agent loop prompt it like a system: clear inputs, verification criteria, deliberate context control. The same tool produces dramatically different results depending on which mental model the developer holds.
</InsightBox>

That difference is visible in a single prompt:


</div>
<div slot="after">

The second prompt names the files, states the symptom, sets a verification bar, and asks to see the change before it lands. It treats Claude as a system with inputs and checks — not a oracle you hope guesses right.

<WorkedExample id="agent-loop-walkthrough" title="The agent loop in one session">

Watch the loop iterate on a real request:

```text
You: "Find the function that handles user authentication."

Claude reasons: "Search the codebase for auth symbols."
Claude calls Grep:        ← tool call (~50 tokens of context)
  pattern: "auth"
  output: 23 matches across 8 files

Claude reasons: "Too many. Narrow it."
Claude calls Grep:        ← tool call (~50 tokens)
  pattern: "def authenticate" OR "def login"
  output: 2 matches in src/auth.py

Claude calls Read:        ← tool call (~400 tokens — the whole file)
  file: src/auth.py

Claude responds: "src/auth.py:47 is authenticate_user().
  It validates credentials and issues a JWT."
```

Each `← tool call` is one turn of the loop. The whole transcript — your prompt, every tool request and result, Claude's reasoning — stays in context until you `/clear` or `/compact`. **A 30-tool-call session can spend half its context budget on tool outputs alone**, which is why "context as currency" becomes a load-bearing idea in Chapter 5.

</WorkedExample>

<Tip n="1" title="Treat Claude as a system, not a chatbot.">
The same tool produces chatbot-quality or system-quality output depending on the mental model you bring to it: clear inputs, observable tool calls, deliberate context.
</Tip>

## Four engineering principles

These four principles underlie every recommendation in this handbook. They apply to code written by AI, code written by humans, and code written by both together.

### 1. Never fail silently

Every error must be explicitly reported with recovery options. Silent failures are the most expensive kind of bug: they produce incorrect results that look correct, propagate through downstream systems undetected, and surface only when the damage is hard to reverse.

<BeforeAfter>
<div slot="before">

```markdown
# CLAUDE.md
Handle errors gracefully.
```

</div>
<div slot="after">

```markdown
# CLAUDE.md
If you encounter an error, report it immediately
with the error message, your analysis of the root
cause, and 2-3 options for resolution.
Never work around errors silently.
```

</div>
</BeforeAfter>

<MarginNote variant="tip">
  The "after" version is longer but produces better outcomes. Precision in error-handling instructions compounds across every session.
</MarginNote>

In AI-assisted development this has an extra dimension: Claude must be _configured_ to surface problems rather than route around them. A vague "handle errors" invites Claude to catch and suppress; an explicit "report errors with analysis" makes every failure visible and actionable.

</Exercise>

### 2. Simplicity over complexity

Short functions. Flat structure. Self-documenting code. A 20-line function with a clear name is superior to a 5-line function with three levels of abstraction.

<InsightBox title="Why simplicity matters more with AI">
  Simple code is easier for Claude to reason about, test, and modify correctly. Each layer of indirection is another opportunity for misunderstanding, so complex abstractions raise the probability of subtle bugs in AI-generated code. And when Claude reads your codebase to learn its conventions, simple patterns propagate accurately while complex ones propagate errors.
</InsightBox>

### 3. Immutability by default

Return new data structures; mark mutations explicitly. Pure functions are easier to test, easier to parallelize, and easier for Claude to reason about.

When mutation is necessary (performance, I/O, state management), make it explicit: name the function `update_`, use mutable types deliberately, and document the side effects. The reader — human or AI — should never be surprised by what a function modifies.

### 4. Fail fast with diagnostics

Stop immediately on problems, with full context: what failed, what was expected, what was received, and what the caller can do about it.

```python
def process_data(df: pd.DataFrame, min_rows: int) -> pd.DataFrame:
    if len(df) < min_rows:
        raise ValueError(
            f"Need {min_rows} rows, got {len(df)}. "
            f"Check data source or reduce min_rows parameter."
        )
    # ... process
```

<MarginNote variant="tip">
  Error messages that include the violation _and_ a suggested fix turn a 30-minute investigation into a 10-second fix.
</MarginNote>

The message says what went wrong (`Need N, got M`), where to look (`data source`), and what to do (`reduce min_rows`). That is the whole distance between a fast fix and a long investigation.

## The codebase is the curriculum

<KeyIdea>
These principles are not Claude-specific — they are engineering principles that become _more_ important when AI generates code. AI amplifies both good and bad patterns. If your codebase follows these principles, Claude's contributions will too. If your codebase is silent about errors, full of mutable state, and deeply abstracted, Claude's contributions will be the same.
</KeyIdea>

This has a direct practical consequence: the single most valuable thing you can do before pointing Claude at an existing project is to make sure the code it will read **exemplifies the patterns you want it to produce**. Configuration reinforces this — your first `CLAUDE.md` (Chapter 2) and best-practice conventions<Citation src="anthropic-claude-code-best-practices" /> sharpen it further — but the codebase itself is the primary teacher.

<Tip n="2" title="The codebase is the curriculum.">
Claude amplifies the patterns it reads, so the highest-leverage preparation is making the code it will see exemplary — before you write a single prompt.
</Tip>

## Common pitfalls

<Pitfall title="Treating Claude as a chatbot">
Vague requests, no verification, no context management — the "before" of every example in this chapter. The fix is the rest of the book, but start here: put verification criteria in your prompt, ask Claude to read before it edits, and `/clear` between unrelated tasks.
</Pitfall>

<Pitfall title="Ignoring tool-call cost">
Letting Claude read 30 files when 3 would do burns context for no signal. Name the files or directories worth looking at up front — Claude is far better at scoping a `Grep` to a directory than at guessing which of 100 files to open first.
</Pitfall>

<Pitfall title="Skipping CLAUDE.md">
Relying on prompt-only configuration. `CLAUDE.md` is the **persistent** layer that survives `/clear`; prompts are session-only. A two-minute investment in a 60-line `CLAUDE.md` pays back every session, forever.
</Pitfall>

## Quick reference

- Claude Code is an agent loop: prompt → reason → tool → observe → repeat.
- The context window is finite — every tool call costs tokens.
- Four principles: never fail silently, simplicity, immutability, fail fast.
- The codebase teaches Claude patterns — good and bad.
- Configuration reinforces good code but does not replace it.

## Practice

<Practice id="ch1-pr-1" difficulty="2">
Pick one function in your codebase that Claude will read often — an entry point, a frequently-imported utility, or a data loader. Audit it against the four principles: does it fail silently? Is it under ~30 lines? Does it mutate its input without saying so? Do its error messages say both what broke and what to do? Fix one issue. This is your first investment in the codebase-as-curriculum.
</Practice>

</Practice>

</Practice>

<ExerciseSolutions>

<Solution for="ch1-ex-1">
Most of the budget goes into the **file reads and test outputs**, not the edit. A `Read` pulls an entire file into context and each test run pulls its full output; the edit itself is a small diff. Implication: scope the next task to the few files that actually matter and capture test output narrowly (or `/clear` between attempts) rather than letting Claude explore freely — the reads, not the writes, are what fill the window.
</Solution>

<Solution for="ch1-ex-2">
Make the failure loud and actionable, for example: _"If the data file is missing, stop and report the path you expected, that it was not found, and 2-3 options — point me at the correct path, regenerate the file, or proceed with an explicitly named default that you call out. Never substitute defaults silently."_ The rewrite turns a silent substitution into a visible decision with recovery options.
</Solution>

<Solution for="ch1-ex-3">
Make the specific files Claude will read most — entry points and frequently-imported utilities — exemplary first. A better prompt can't override what the surrounding code consistently teaches: Claude amplifies the patterns it reads, so inconsistent error handling in the files it learns from will resurface in its output no matter how the prompt is worded. Fix the curriculum, then prompt.
</Solution>

</ExerciseSolutions>
