_A code review needs four perspectives: correctness, security, performance, and style. Run them sequentially and it takes twenty minutes; run them in parallel and it takes five. But the win is not only speed. A reviewer who cannot see the writer's reasoning catches different bugs. This chapter is about delegating to subagents as a developer ‚Äî when it pays, how to task them well, and how to keep parallel work from colliding. The orchestration **layer** that coordinates many agents at scale is its own chapter._

## Subagents: isolated context workers

A **subagent** is a purpose-built Claude instance that runs in its own context, isolated from your main session.<Citation src="claude-code-overview" /> It receives a task, executes it with its own tool set, and returns a result to the parent ‚Äî the parent's context never fills with the subagent's file reads and dead ends. That isolation is the whole point: a subagent that explores ten files and reports two findings costs you two findings of context, not ten files'.

You define a custom subagent as a Markdown file under `.claude/agents/`, with frontmatter on top and a system prompt below.

```markdown
# .claude/agents/data-quality-reviewer.md
---
name: data-quality-reviewer
description: Review a data pipeline for quality issues
tools: [Read, Grep, Glob]
---
You are a data quality specialist. Review the provided
pipeline code for:
1. Schema violations and type mismatches
2. Null handling (silent drops vs. explicit imputation)
3. Distribution drift between train and serve
4. Data leakage between train and test splits
Report findings with severity ratings (Critical/High/Medium/Low).
```

Note the frontmatter key is `tools` ‚Äî **not** `allowed-tools`. Skills use `allowed-tools` (hyphenated); subagents use `tools` (no hyphen). The two surfaces follow different specification standards, and the inconsistency is deliberate, not a typo.<Citation src="claude-code-overview" />

<MarginNote variant="note" label="Note">
  **Subagent** here means one isolated worker with its own context. An **agent team** ‚Äî a collaborative group that shares a task list and messages each other ‚Äî is a different surface, covered in Ch 11.
</MarginNote>

### The frontmatter keys

| Key | Purpose |
|---|---|
| `name` | Display name ‚Äî lowercase, hyphens. |
| `description` | What the agent does; used for auto-dispatch. |
| `tools` | Tools the agent may use (note: *not* `allowed-tools`). |
| `model` | Override the model (`sonnet`, `opus`, `haiku`). |
| `isolation` | Set to `worktree` for an isolated file system (see below). |
| `memory` | Persistent memory scope: `user`, `project`, or `local`. |
| `background` | Run in the background; the parent is notified on completion. |
| `skills` | Skills available to the subagent. |
| `mcpServers` | MCP servers the subagent can access. |

Manage these configurations interactively with the `/agents` command.<Citation src="claude-code-overview" />

### Built-in subagent types

You do not always need a custom definition. Claude Code ships several built-in subagent types you can invoke through the **Task tool** directly:

- **Explore** ‚Äî fast codebase exploration: find files, search code, answer "where is X?"
- **Plan** ‚Äî read-only analysis for implementation planning.
- **General-purpose** ‚Äî a multi-tool agent for complex, open-ended tasks.

In practice you rarely type "use the Explore subagent." You say *"find every place we parse the auth token"* and Claude dispatches an Explore agent; the result returns as a summary, not as ten files in your context.

### Continuing a subagent's work

Each new invocation of the Task tool starts a **fresh** subagent with no memory of a previous one. Resuming one with its context intact ‚Äî asking Claude to *"continue the review agent"* ‚Äî is possible, with two caveats: the built-in **Explore and Plan agents are one-shot** (they return no agent ID, so they can't be resumed ‚Äî use `general-purpose` or a custom subagent when you need to continue), and the underlying resume mechanism (the `SendMessage` tool) is currently **available only when agent teams are enabled** (`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`).<Citation src="anthropic-claude-code-sub-agents" /> Outside that, treat each new invocation as a fresh start.

<Tip n="37" title="A subagent buys you context isolation ‚Äî spend it on heavy exploration.">
The value of a subagent is that its file reads, searches, and dead ends never touch your main session. Delegate the work that would otherwise bloat your context; keep the work that needs your context in your context.
</Tip>

## When to delegate

Delegation is not free ‚Äî every subagent spends its own context tokens ‚Äî so the decision is real. The test is about *dependency* and *size*, not difficulty.

**Delegate when:**

- The task is **independent** of your current work ‚Äî it does not need what you are holding in your session.
- It requires **heavy context** that would otherwise pollute your main session (a wide codebase sweep, a long log).
- Multiple tasks can run in **parallel** for a wall-clock saving.
- You want an **unbiased second opinion** ‚Äî the Writer/Reviewer pattern below.

**Do it directly when:**

- The task takes **under two minutes** ‚Äî the agent's startup overhead exceeds the saving.
- It **depends on your session's context** (what you just discussed, the file you just edited).
- **Sequential reasoning** is required ‚Äî step B needs step A's result.
- It is simply **too small** to justify the overhead.

<MarginNote variant="note" label="Cost">
  Each subagent consumes its own context tokens. Four parallel agents cost roughly 4√ó the tokens but take only 1√ó the wall-clock time. Parallelism trades cost for speed ‚Äî spend it deliberately.
</MarginNote>

## Designing effective agent tasks

Knowing *when* to delegate is necessary but not sufficient. The quality of the delegation determines the quality of the result ‚Äî and the most common failure is vague tasking.

### The three properties of a good agent task

1. **Self-contained.** The agent must complete the task without coming back to ask the parent for clarification. If it needs context from your session, either pass that context explicitly in the prompt or do the task yourself.
2. **Verifiable.** The output must be checkable. *"Review this code"* produces opinions; *"run the test suite and report failures with file:line references"* produces verifiable facts. Prefer tasks with concrete success criteria.
3. **Bounded.** An agent told to find *"everything wrong with this codebase"* will burn its whole context window and return an unfocused report. Constrain the scope: *"review `src/auth/` for SQL-injection vulnerabilities."*

<KeyIdea>
Specificity in the prompt determines specificity in the output. "Review this PR" yields generic feedback. "Review `src/payments/` for (1) race conditions in concurrent checkout, (2) floating-point precision in amount calculations, (3) missing error handling on gateway timeouts" yields actionable findings. The agent cannot be more precise than your tasking.
</KeyIdea>

### When parallel execution actually helps

Parallel agents save wall-clock time **only when the tasks are truly independent**. The test is one question per pair:

- Can Agent B start without knowing Agent A's result? ‚Üí **Parallel.**
- Does B's work depend on A's output? ‚Üí **Sequential.** Run A, feed its result to B.
- Do both agents modify the same files? ‚Üí **Use worktrees**, or make one read-only.

Parallelism has overhead. Each agent loads its own context, reads files independently, and may duplicate exploration the others are also doing. For tasks under two minutes, that overhead exceeds the saving.

<InsightBox title="The parallelism sweet spot is narrow">
  The sweet spot is **2‚Äì4 agents on tasks that each take 5‚Äì15 minutes**. Below ~2 minutes per task, the per-agent startup overhead dominates and sequential is faster. Above 4 agents, coordination cost grows and quality degrades ‚Äî not because the agents are worse, but because *you* cannot review four-plus reports carefully enough to trust them. The bottleneck moves from compute to your attention.
</InsightBox>

### Designing agent prompts

1. **State the role**, not just the task. *"You are a security specialist"* focuses attention differently than *"review this code."*
2. **Specify the output format.** *"Report as: file:line | severity | description"* produces structured, comparable results across agents ‚Äî essential when you run several and need to merge their findings.
3. **Tell the agent what to ignore.** *"Do not report style issues ‚Äî focus only on correctness"* removes the noise that buries real findings.
4. **Restrict tools to what's needed.** A review agent needs `Read, Grep, Glob` ‚Äî not `Edit` or `Bash`. Fewer tools mean fewer distractions and a smaller blast radius.

<Tip n="38" title="Make every agent task verifiable and bounded.">
Give the agent a concrete success criterion and a scope it can finish inside one context window. "Run the tests and report failures" beats "check the code"; "review `src/auth/`" beats "find everything wrong."
</Tip>

## Git worktrees: isolated file systems

Two agents editing the same files will collide. A **git worktree** solves this by giving each agent its own complete, isolated copy of the repository ‚Äî its own working tree on its own branch, sharing the underlying git objects.<Citation src="anthropic-claude-code-common-workflows" />

You can manage worktrees by hand, or let Claude manage their lifecycle for you:

<BeforeAfter title="Manual worktree management vs. the --worktree flag">
<div slot="before">

```bash
# Manual: you create the worktree and remember to clean it up
git worktree add ../feature-cache -b feature-cache
cd ../feature-cache && claude
# ... work ...
cd - && git worktree remove ../feature-cache
```

</div>
<div slot="after">

```bash
# Claude manages the worktree lifecycle ‚Äî pass a name
# (or omit it for an auto-generated one), then work in that session
claude --worktree feature-cache
claude --worktree rate-limiting
```

</div>
</BeforeAfter>

Each worktree gets its own branch and a complete isolated copy of the repo. Cleanup is automatic: if no changes were made, the worktree is removed silently; if changes exist, you are prompted to merge or discard.

A subagent can run in its own worktree too, via the `isolation` frontmatter key:

```markdown
# .claude/agents/parallel-reviewer.md
---
name: parallel-reviewer
tools: [Read, Grep, Glob]
isolation: worktree
---
Review the code for correctness and edge cases.
Report findings with file:line references.
```

With `isolation: worktree` set, the subagent operates on its own copy of the repository ‚Äî essential for any agent that *modifies* files, since two agents editing the same file without isolation will conflict.<Citation src="claude-code-overview" />

<MarginNote variant="warning" label="Warning">
  Add `.claude/worktrees/` to `.gitignore` ‚Äî Claude stores worktree metadata there and it should not be committed. If orphaned worktrees pile up after a crash, run `git worktree prune` to clear the stale entries.
</MarginNote>

<MarginNote variant="note" label="Cost">
  Worktrees trade disk for isolation. They share git objects but duplicate the working tree, so a 500 MB repo spawning four worktrees may use ~1 GB extra. `git worktree prune` reclaims stale ones.
</MarginNote>

<Tip n="39" title="Isolate file-modifying agents in worktrees; leave read-only agents shared.">
If two agents will write, give each a worktree (`--worktree`, or `isolation: worktree`). If they only read ‚Äî reviewers, explorers ‚Äî they can safely share the tree. Match isolation to whether the agent has a writing tool at all.
</Tip>

## The Writer/Reviewer pattern

Use **separate sessions** to defeat confirmation bias.<Citation src="anthropic-claude-code-best-practices" />

1. **Writer session** ‚Äî makes changes against the requirements.
2. **Reviewer session** ‚Äî reviews the *diff* with no knowledge of intent: *"Review this diff for bugs, regressions, and missed edge cases. You have no context about why these changes were made."*

This catches errors the writer is blind to precisely because the writer has accumulated context about the "correct" approach ‚Äî and that context is exactly what makes it stop questioning its own work. The reviewer, starting cold, has no such commitment.

You can extend the same idea to multi-perspective review by running several reviewer subagents in parallel, each with one lens:

1. **Correctness** ‚Äî "Does this logic do what it claims?"
2. **Data quality** ‚Äî "Schema violations? Null patterns? Drift?"
3. **Statistical validity** ‚Äî "Are the assumptions met? Distributions stable?"
4. **Reproducibility** ‚Äî "Seeds set? Configs versioned? Results deterministic?"

Run four in parallel and it costs the same wall-clock time as one ‚Äî and you get four independent reads instead of one writer's self-assessment. Think *together* first (Chapter 6), then *separate* for independent review.

<KeyIdea>
Parallelism in Claude Code is not about speed alone ‚Äî it is about **perspective isolation**. A reviewer that cannot see the writer's reasoning catches different bugs; a security agent that sees only the code, not the intent, finds different vulnerabilities. The diversity of independent perspectives is the value, with the time saving as a bonus.
</KeyIdea>

## The /batch skill: parallel codebase changes

`/batch` is a bundled skill (not a built-in command) that orchestrates a large-scale change across a codebase in parallel.<Citation src="claude-code-overview" /> It researches the codebase, decomposes the work into **5‚Äì30 independent units**, presents a plan for your approval, then spawns **one background agent per unit in its own git worktree**. Each agent implements its unit, runs the tests, and opens a pull request.

```text
/batch migrate src/ from class components to hooks
/batch add type hints to all Python files in lib/
/batch update all API endpoints to return JSON:API format
```

The plan-review step is your checkpoint: test on two or three units before approving a twenty-unit run, because a twenty-unit batch costs roughly twenty times a single session.

When to reach for `/batch` versus a fan-out script (Chapter 13's `for f in ‚Ä¶; claude -p ‚Ä¶` loop):

| | `/batch` | Fan-out script |
|---|---|---|
| **Who decomposes** | Claude researches and splits the work | You define the units up front |
| **Best for** | Complex changes where decomposition needs understanding the codebase | Simple, repetitive changes where you already know the file list |
| **Output** | one PR per unit, each in its own worktree | whatever your script does per file |

<MarginNote variant="note" label="Cost">
  `/batch` spawns N agents √ó per-agent context ‚Äî a 20-unit batch costs ~20√ó a single session. The initial plan review is your one chance to scope it down before the spend.
</MarginNote>

## Common pitfalls

<Pitfall title="Vague tasking">
"Review this PR" produces generic feedback because the agent cannot be more specific than your prompt. State the role, scope the target, list what to look for, and say what to ignore. Specificity in equals specificity out.
</Pitfall>

<Pitfall title="Parallelizing dependent work">
If Agent B needs Agent A's output, running them in parallel just means B works from stale or missing information. Apply the test ‚Äî *can B start without A's result?* ‚Äî before you fan out. Dependent steps run sequentially, full stop.
</Pitfall>

<Pitfall title="Unisolated file modifications">
Two agents editing the same files without worktrees will overwrite each other and leave a corrupted, half-merged tree. Any agent with a writing tool that runs alongside another writer needs `isolation: worktree` (or `--worktree`); read-only agents can share.
</Pitfall>

<Pitfall title="Over-fanning out">
More than four parallel agents rarely helps: coordination overhead climbs and you cannot review every report carefully, so you end up trusting output you did not actually verify. Stay in the 2‚Äì4 / 5‚Äì15-minute band; the limit is your attention, not the compute.
</Pitfall>

<Pitfall title="Expecting a re-invoked agent to remember">
A new Task invocation starts a fresh agent with no memory of the last one. Resuming with accumulated context works only where it's supported ‚Äî agent teams enabled, and not the one-shot Explore/Plan agents. Where it isn't, don't assume "run the review agent again" continues the prior run; it starts over.
</Pitfall>

## Quick reference

- **Subagents**: isolated context, independent tool set, return a summary to the parent. Defined in `.claude/agents/*.md`; managed with `/agents`.
- **Frontmatter keys**: `tools` (note: *not* `allowed-tools` ‚Äî skills use the hyphenated form), `model`, `isolation`, `memory`, `background`, `skills`, `mcpServers`.
- **Built-in types** via the Task tool: **Explore**, **Plan**, **Bash**, **General-purpose**.
- **Resuming** a subagent keeps its context (Explore/Plan are one-shot; resume needs agent teams enabled) ‚Äî otherwise a new invocation starts fresh.
- **Delegate when**: independent, context-heavy, parallelizable, needs a fresh perspective. **Do directly when**: under two minutes, context-dependent, sequential, or trivial.
- **Good agent task**: self-contained, verifiable, bounded.
- **Parallelism sweet spot**: 2‚Äì4 agents, 5‚Äì15-minute tasks. Under ~2 min, overhead dominates; over 4, coordination + review attention degrade.
- **Worktrees**: isolated file systems for parallel writes ‚Äî `claude --worktree "‚Ä¶"` or `isolation: worktree`. Gitignore `.claude/worktrees/`; `git worktree prune` cleans stale entries.
- **Writer/Reviewer**: separate sessions defeat confirmation bias; extend to multi-perspective reviewers in parallel.
- **`/batch`**: Claude decomposes into 5‚Äì30 units, one background agent per unit in its own worktree, each opens a PR. Use the plan review as your scope-down checkpoint.

<Solution for="ch10-ex-1">
In the agent loop, every file the model reads becomes part of the context window it reasons over for the rest of the session, and that window is finite (Chapter 1). Done inline, a twenty-file review would load twenty files' worth of tokens into *your* session and they would stay there ‚Äî crowding out the work you actually care about and pushing you toward compaction. A subagent runs the same loop in a *separate* finite window: it reads the twenty files into its own context, reasons, and returns only the four-line summary. You pay four lines of context for a twenty-file review. That trade ‚Äî pay for the conclusion, not the exploration ‚Äî is the entire reason subagents exist.
</Solution>

<Solution for="ch10-ex-2">
(a) the rename is the *borderline* case and (b) the auth audit is the *good* delegation candidate ‚Äî and the deciding property is context-heaviness, not difficulty. The rename is mechanical and may touch your current files, so it can be context-dependent and is often fast enough that the agent's startup overhead is not worth it (do it directly, or fan it out only if it spans many files you are *not* touching). The auth audit is independent of whatever you are doing, requires a wide read across `src/auth/` that would bloat your main session, and has a verifiable, bounded output ("missing rate-limit checks, with file:line") ‚Äî exactly the heavy-context, independent, checkable profile that delegation is for.
</Solution>

<Solution for="ch10-ex-3">

Use only Read, Grep, and Glob."_ ‚Äî Naming the directory and the three specific concerns makes it **bounded** (it fits one context window instead of "the API layer" sprawling everywhere). The fixed report format and the concrete list of what to look for make it **verifiable** (you can check each finding against a line). Stating the role, the format, and what to ignore ‚Äî plus restricting the tools ‚Äî makes it **self-contained**: the agent has everything it needs and never has to come back to ask what you meant by "concerning."
