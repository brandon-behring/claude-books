## Security and compliance

A security review opens with one question: which certifications do you hold? Claude Code is built on Anthropic's security program; the first-party security documentation names **SOC 2 Type 2** and **ISO 27001**, with reports and certificates available at the Anthropic Trust Center.<Citation src="anthropic-claude-code-security" /> The Enterprise offering adds a **HIPAA-ready** deployment, and further attestations (such as ISO/IEC 42001) are published alongside them.<Citation src="anthropic-claude-enterprise" />

<KeyIdea>
Don't paste a certification list into a deployment doc — cite the [Trust Center](https://trust.anthropic.com). The list changes as new attestations land, and the Trust Center is both the authoritative, dated source and the artifact a security reviewer will actually accept. A frozen list in a wiki is stale the day an audit renews.
</KeyIdea>

<Tip n="43" title="Cite the Trust Center; don't freeze the cert list.">
Certifications renew and expand on their own schedule. Point reviewers to trust.anthropic.com — the dated system of record — rather than a copied list that silently goes stale.
</Tip>

### Identity and access

For organization rollouts, Claude's Enterprise plan provides the identity controls IT expects:<Citation src="anthropic-claude-enterprise" />

- **Single sign-on (SSO)** — on Team and Enterprise.
- **SCIM provisioning** — automated user lifecycle (create / update / deprovision) on Enterprise.
- **Role-based access with fine-grained permissioning** — on Enterprise.
- **Domain capture** — claim your email domain so accounts join the org, on Team and Enterprise.
- **Audit logs and a Compliance API** for observability and monitoring.

### Data protection

Session data travels over **TLS**, and Enterprise adds **custom data-retention controls**.<Citation src="anthropic-claude-code-security" /><Citation src="anthropic-claude-enterprise" /> On training: consumer tiers (Free, Pro, Max) are **opt-in** — training is off unless a user enables it, and disabling it stops future use — while commercial products (the API, Team, Enterprise) are governed by the Commercial Terms rather than that consumer setting.<Citation src="anthropic-claude-code-security" /> Zero Data Retention is available for commercial and API use; confirm the current terms in the commercial privacy documentation before you cite a specific guarantee.

<MarginNote variant="note" label="Note">
  The consumer training toggle (opt-in, de-identified, up to five years if enabled) is a *consumer* setting. Don't assume it describes how Team/Enterprise/API data is handled — those follow the Commercial Terms, where the default is not-trained-on.
</MarginNote>

## Government and regulated workloads

Claude reaches the highest US-government bars — but not as Claude Code itself. Anthropic's models receive **FedRAMP High** and **DoD Impact Level 4 and 5** authorization through **Amazon Bedrock in AWS GovCloud (US)**, and reach FedRAMP-High workloads through **Google Vertex AI with Assured Workloads**.<Citation src="aws-bedrock-fedramp-govcloud" /> The authorization is the *platform's*: the models operate as software components inside an already-authorized service, inheriting its boundary.

<Pitfall title="Saying 'Claude Code is FedRAMP High'">
It isn't — and a government security reviewer will catch it. FedRAMP High and DoD IL-4/5 are authorizations of the **hosting platform** (Bedrock in AWS GovCloud, or Vertex AI Assured Workloads), which the models inherit as components. Claude Code and the Claude API are not themselves FedRAMP-authorized. Phrase it as "Claude via Bedrock GovCloud," and route regulated workloads through that platform.
</Pitfall>

<Tip n="44" title="FedRAMP is the platform's, not Claude Code's.">
Government authorization comes through Bedrock GovCloud or Vertex Assured Workloads, which the models inherit. Say "via Bedrock GovCloud," never "Claude Code is FedRAMP High."
</Tip>

## Governance at organization scale

The team controls from [Chapter 15](/chapters/ch15-team-patterns) become the enterprise governance surface. Access control is delivered through **managed (policy) settings**, not a separate product — they sit at the top of the [settings precedence](/chapters/ch09-scaling-configuration) and can't be overridden locally:<Citation src="anthropic-claude-code-settings" />

- `availableModels` + `enforceAvailableModels` — restrict the selectable models org-wide (and constrain the Default to the allowlist).
- `requiredMinimumVersion` — block startup on outdated clients.
- `disableWorkflows` — turn off dynamic workflows org-wide.
- `allowManagedHooksOnly` — load only managed/SDK hooks.
- `disableRemoteControl` — disable Remote Control at the device level.

Admin toggles at `claude.ai/admin-settings/claude-code` govern Remote Control and Dynamic Workflows centrally; **OpenTelemetry** metrics, **audit logs**, a **Compliance API**, and **admin-set spend limits** (per user and per org) round out the monitoring surface.<Citation src="anthropic-claude-enterprise" />

## Cost optimization at scale

At fifty seats the bill is seat-price plus usage, and the instinct to "just use a cheaper model everywhere" is usually the wrong first move. Reach for the levers in this order:

1. **Prompt caching** — cache reads run at roughly **90% off** the input price. The single biggest lever, and it keeps the model you actually want.
2. **Batch API** — **50% off** for asynchronous, non-interactive work (overnight test generation, bulk analysis).
3. **Model right-sizing** — Sonnet for the majority of coding work, Opus for the hardest reasoning, Haiku for simple/cheap tasks. Right-size per task, don't downgrade globally.
4. **Spend caps** — admins set per-user and per-org limits as a backstop.

<KeyIdea>
Caching beats downgrading. A team that turns on prompt caching keeps Opus-or-Sonnet quality at a fraction of the token cost; a team that downgrades everyone to the cheapest model to save money pays in slower, lower-quality work that costs more developer time than it saves in tokens. Cache first; right-size second; downgrade only where the task genuinely doesn't need the capability.
</KeyIdea>

By default an **Enterprise subscription seat** runs **Sonnet 4.6** and **Enterprise pay-as-you-go** runs **Opus 4.8** — set the org allowlist deliberately rather than inheriting the default.<Citation src="anthropic-models-overview" />

## A phased 50-person rollout

Don't switch fifty developers on at once. Phase it so each step is independently valuable and reversible — the same incremental discipline the rest of the book applies to configuration:

| Phase | Who | What |
|---|---|---|
| **Week 1 — Pilot** | 5 developers | Stand up shared `CLAUDE.md` + managed settings; pick one real project; gather friction. |
| **Week 2 — Expand** | 15 developers | Add the skills/commands the pilot proved; onboard with a day-one checklist. |
| **Week 3 — Scale** | 50 developers | Roll out org-wide; turn on monitoring (OpenTelemetry, audit logs); set spend caps. |
| **Week 4 — Optimize** | all | Review usage; enable caching everywhere; right-size the model allowlist; trim `CLAUDE.md`. |

## Deployment decision matrix

Not every need is a Claude Code need. Match the surface to the job:

| Need | Reach for |
|---|---|
| Interactive development with team coding standards | **Claude Code** |
| Quick questions, document analysis, ad-hoc help | **Claude apps** (web / desktop) |
| Automation, bulk processing, custom applications | **API + Agent SDK** |
| Compliance, SSO/SCIM, governed rollout | **Enterprise plan** |

<Tip n="45" title="Cache before you downgrade.">
The cheapest way to cut a large bill is prompt caching (~90% off cache reads), not a global move to a weaker model. Cache first, right-size per task second, and downgrade only where the work genuinely doesn't need the capability.
</Tip>

## Common pitfalls

<Pitfall title="Claiming Claude Code itself is FedRAMP authorized">
The authorization belongs to the hosting platform (Bedrock GovCloud, Vertex Assured Workloads); the models inherit it as components. Misstating this in a government procurement doc is the kind of error that stalls an approval. Route regulated workloads through the authorized platform and describe it as such.
</Pitfall>

<Pitfall title="Freezing the certification list">
A cert list copied into a doc is stale the moment an attestation renews or a new one lands. Point to the Trust Center — it's dated, authoritative, and what reviewers accept.
</Pitfall>

<Pitfall title="Downgrading the model before turning on caching">
Moving everyone to the cheapest model to save money trades token cost for developer time and quality. Prompt caching is the larger lever and keeps the capability; reach for it first.
</Pitfall>

<Pitfall title="Big-bang rollout to the whole org">
Switching fifty developers on at once means no pilot to surface friction and no safe rollback. Phase it — five, then fifteen, then fifty — and turn on monitoring before the final wave.
</Pitfall>

<Pitfall title="Assuming the consumer training toggle covers commercial data">
The opt-in training setting is a consumer-tier control. Team/Enterprise/API data is governed by the Commercial Terms, where the default is not-trained-on — cite the Commercial Terms, not the consumer privacy setting, when answering a data-handling question.
</Pitfall>

## Quick reference

- **Certifications**: SOC 2 Type 2 + ISO 27001 (first-party); HIPAA-ready (Enterprise); full current list at **trust.anthropic.com** — cite it, don't freeze it.
- **IAM (Enterprise)**: SSO, SCIM provisioning, role-based access with fine-grained permissioning, domain capture, audit logs, Compliance API.
- **Government**: FedRAMP High + DoD IL-4/5 **via Bedrock GovCloud**; FedRAMP-High via **Vertex Assured Workloads** — platform authorization, not Claude-direct.
- **Governance**: managed settings (`availableModels`/`enforceAvailableModels`, `requiredMinimumVersion`, `disableWorkflows`, `allowManagedHooksOnly`, `disableRemoteControl`) + admin toggles + OpenTelemetry + audit logs + spend caps.
- **Cost levers, in order**: prompt caching (~90% off reads) → Batch API (50%) → model right-sizing → spend caps. Pricing is a dated snapshot — verify at claude.com/pricing.
- **Rollout**: pilot 5 → expand 15 → scale 50 → optimize.
