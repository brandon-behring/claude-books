_You keep running the same four commands by hand — format, lint, test, commit. You re-explain the same deployment procedure every session. There must be a way to encode these workflows so Claude runs them without being told each time. Claude Code ships four extension mechanisms — commands, skills, hooks, and MCP servers — that form a ladder of increasing autonomy. Knowing **which rung to reach for** is the difference between a tool and a workflow, and it is the lever this chapter is built around._

## Choosing the right mechanism

Before the mechanics of any one extension, answer the strategic question: _given a workflow problem, which extension do you reach for?_ Getting this wrong costs more than the implementation — it costs ongoing maintenance on the wrong abstraction.

The four mechanisms form a ladder. Each rung adds capability **and** setup cost; the simplest rung that solves the problem is the right one.

<KeyIdea>
**The extension ladder — climb only as far as the problem forces you.**

1. **Command** — a Markdown file that expands into a prompt. Zero infrastructure. _Reach for it when:_ the workflow is a single prompt that needs a shortcut (`/check`, `/deploy`).
2. **Skill** — a prompt-driven playbook Claude executes with tool access; can read files, branch, spawn subagents. _Reach for it when:_ the work is multi-step or needs domain knowledge and Claude must _decide_ what to do next (code review, release checklist).
3. **Hook** — deterministic automation on a lifecycle event. Claude does not decide whether it runs — the harness does. _Reach for it when:_ something must happen **every time, without exception** (auto-format, lint-on-edit, security scan on commit).
4. **MCP server** — structured access to an external service with schema-defined inputs and outputs. _Reach for it when:_ Claude must talk to a database, an API, or an internal tool with typed I/O (Jira, Postgres, Slack).

Read the axis as: **capability increases as you climb; so does setup and maintenance cost.** Start at the bottom; promote only when you hit a rung's limit.
</KeyIdea>

<MarginNote variant="tip" label="Tip">
  Start with the simplest mechanism. Promote only when you hit its limits — never the other way round.
</MarginNote>

### Common mistakes

The four classic misfires all spend effort one rung too high (or one too low):

1. **Building a skill when a command suffices.** If the workflow is a fixed sequence with no branching, a command is simpler to maintain. Skills earn their complexity only when Claude needs to _decide_ what to do next.
2. **Using a skill when a hook is needed.** Skills are _advisory_ — Claude can ignore or adapt them. If the behavior must be _enforced_ (security scan, formatting), use a hook. Hooks run whether or not Claude remembers to invoke them.
3. **Reaching for MCP when a CLI exists.** Claude already knows `gh`, `aws`, `gcloud`, `psql`. MCP is for services without a good CLI, or when you need structured I/O schemas.<Citation src="anthropic-claude-code-best-practices" />
4. **Encoding team standards in skills instead of hooks.** "Every PR must pass linting" belongs in a `PreToolUse` hook on `Bash(git commit*)`, not a skill someone might forget to invoke.

### The promotion path

Extensions evolve naturally — and the right move is usually to let them, not to skip ahead:

1. You type the same prompt three times → extract it into a **command**.
2. The command needs conditional logic ("if tests fail, do X") → promote it to a **skill**.
3. The skill must run on _every_ commit, not just when invoked → extract the enforcement part into a **hook**.
4. The hook needs data from an external service → add an **MCP server** and call it from the hook.

<InsightBox title="The real failure mode is skipping the command stage">
  Choosing the _wrong_ rung is rarely the problem. The common failure is jumping straight to a complex skill or MCP server when a three-line command file would have solved it for now. Most workflows that end up as elaborate skills started life as a prompt someone typed repeatedly. If you find yourself building a skill on day one, pause: would a command do for now? A command handles the large majority of workflow needs; add skills, hooks, and MCP only when the simpler rung is genuinely insufficient.
</InsightBox>

<Tip n="34" title="Climb the extension ladder; don't leap it.">
Reach for the lightest mechanism that solves the problem — command before skill, skill before hook, hook before MCP. Each rung up buys capability at the price of maintenance; only step up when the rung below can't express what you need.
</Tip>

## Custom slash commands

Commands are the lightest rung: Markdown files that expand into a prompt when invoked.<Citation src="anthropic-claude-code-skills" /> Place them in `~/.claude/commands/` (global) or `.claude/commands/` (project), and invoke as `/command-name`. They accept arguments via `$ARGUMENTS` and reference files with `@path`.

```markdown
# .claude/commands/check.md
Run a comprehensive check of the project:
1. Run the test suite: `pytest tests/`
2. Run the linter: `ruff check src/`
3. Run the type checker: `mypy src/`
4. Report results as a summary table
```

The best commands aren't shortcuts — they encode _decisions_. Instead of a command that runs a test, write one that decides _what to do next_ based on project state:

```markdown
# .claude/commands/next.md
Check the current project state:
1. Read CURRENT_WORK.md for context
2. Run `git status` and `git diff --stat`
3. Check for failing tests
4. Recommend the single highest-priority next action
```

<MarginNote variant="note" label="Official">
  Custom slash commands have been merged into skills: both <code>.claude/commands/</code> and <code>.claude/skills/</code> work, and skills take precedence if names conflict.
</MarginNote>

## Skills: domain knowledge and workflows

Skills combine domain knowledge with executable workflows.<Citation src="anthropic-claude-code-skills" /> Each skill lives in its own directory as a `SKILL.md` file. Unlike commands (prompt templates), a skill can be applied automatically when Claude detects a relevant context, define multi-step procedures with validation gates, restrict tools via `allowed-tools`, run shell commands before its content reaches Claude, and spawn a forked subagent context.

### File structure and precedence

```text
.claude/skills/qa-health/SKILL.md      # Project skill
~/.claude/skills/deploy-check/SKILL.md # Personal (all projects)
```

Precedence runs **enterprise (managed settings) > personal > project**. Plugin skills use a namespaced form: `plugin-name:skill-name`.

### Frontmatter reference

All keys are optional; `name` defaults to the directory name.

| Key | Purpose |
|---|---|
| `name` | Display name and `/slash-command`. Lowercase, hyphens, ≤64 chars. |
| `description` | What the skill does. Claude reads this to decide when to auto-apply. |
| `allowed-tools` | Tools permitted without prompting while the skill is active. **Hyphenated.** |
| `argument-hint` | Autocomplete hint, e.g. `[issue-number]`. |
| `model` | Override model while active (`sonnet`, `opus`, `haiku`). |
| `context` | Set to `fork` to run in a forked subagent context. |
| `agent` | Which subagent type to use when `context: fork` is set. |
| `hooks` | Lifecycle hooks scoped to this skill's execution. |
| `user-invocable` | Set `false` to hide from the `/` menu. Default `true`. |
| `disable-model-invocation` | Set `true` to prevent Claude auto-loading. Default `false`. |

### A basic skill

```yaml
---
name: qa-health
description: Run quality dashboard across all volumes
allowed-tools: [Bash, Read, Glob]
---
# QA Health Check
1. Run `python scripts/audits/health_dashboard.py`
2. Parse output for RED/YELLOW/GREEN status
3. Report findings with actionable recommendations
4. If any RED items, create a TODO list for fixes
```

A skill can also execute shell commands _before_ its content reaches Claude, injecting live project state into the prompt — the `` !`command` `` syntax runs at skill-load time:<Citation src="anthropic-claude-code-skills" />

```yaml
---
name: review-changes
description: Review recent changes with full context
allowed-tools: [Bash, Read, Grep]
---
# Review Changes
Current branch status:
!`git status --short`

Review the changes above for logic errors, missing tests,
and style violations per our CLAUDE.md.
```

Including the word "ultrathink" anywhere in a skill's content enables extended thinking for that skill's run.

### Forked context: skill as subagent

For skills that need isolation — long-running analysis, exploratory research, or work that shouldn't pollute your main context — set `context: fork`. The forked context runs as a subagent: it gets its own context window, doesn't consume your main session's context, and returns only a summary when done. This is the bridge between skills and subagents (the parallel-work patterns live in [**Chapter 10 — Agents & Parallel Work**](/chapters/ch10-agents-parallel)).

```yaml
---
name: deep-audit
description: Run thorough code audit in isolated context
context: fork
agent: general-purpose
model: opus
allowed-tools: [Read, Grep, Glob, Bash]
---
# Deep Audit
Analyze the entire codebase for security vulnerabilities,
performance bottlenecks, and dead code. Report findings with
file:line references and severity.
```

### Budget and permissions

Skill descriptions consume roughly 2% of the context window (fallback: 16,000 characters); run `/context` to see whether any skills were excluded.<Citation src="anthropic-claude-code-skills" /> Permission rules can target skills specifically:

```json
{
  "permissions": {
    "allow": ["Skill(qa-health)"],
    "deny": ["Skill(deploy *)"]
  }
}
```

`Skill(name)` matches exactly; `Skill(name *)` matches a prefix with any arguments; bare `Skill` denies all skill usage.

<Tip n="35" title="Measure a skill's value by frequency × savings.">
A skill that saves 5 minutes per use and runs 3 times a week saves ~13 hours a year. If a candidate skill doesn't clear that bar, a command probably suffices.
</Tip>

## Hooks: deterministic automation at a glance

Hooks are the **enforcement layer**. Unlike `CLAUDE.md` instructions (which are advisory), hooks _always execute_ — Claude doesn't choose whether to run one; the harness fires it on a lifecycle event. Configure them via `/hooks` or in `settings.json`.<Citation src="anthropic-claude-code-hooks" />

### Handler types

A hook event triggers one of four handler types. (Not all events support all four — some are command-only.) The command timeout is 10 minutes; HTTP defaults to 30 seconds.<Citation src="anthropic-claude-code-hooks" />

- **`command`** — a shell script; receives JSON on stdin, returns JSON on stdout. The original and most common type.
- **`http`** — POSTs JSON to a URL. Best for webhooks, external notifications, CI triggers.
- **`prompt`** — a single-turn LLM evaluation returning yes/no. Best for policy checks that need judgment, not just pattern matching.
- **`agent`** — a multi-turn subagent with full tool access. Best for complex validation that reads files, runs commands, or reasons across steps.

### The high-traffic events

Claude Code exposes about **29 lifecycle hook events** as of June 2026 (this is the Claude Code `settings.json` surface — the Agent SDK's hook surface is a separate, smaller set; don't conflate them).<Citation src="anthropic-claude-code-hooks" /> Day to day, a handful carry almost all the traffic:

| Event | Fires when | Typical use |
|---|---|---|
| `SessionStart` | a session begins or resumes | load context, run a health check |
| `UserPromptSubmit` | the user submits a prompt | transform or validate input |
| `PreToolUse` | before a tool call runs | validate inputs, block operations |
| `PostToolUse` | after a tool call succeeds | auto-lint, log, trigger follow-up |
| `Stop` | Claude finishes responding | run cleanup |
| `PreCompact` | before context compaction | save state |
| `SessionEnd` | a session ends | persist state, update logs |

For the full enumeration — all ~29 events, their matcher values, and which handler types each supports — see _Appendix C → the full hook-event reference (forthcoming)_.

### Notification hooks

The `Notification` event fires when Claude needs human attention — ideal for a desktop alert during a long unattended run:

```json
{
  "hooks": {
    "Notification": [{
      "hooks": [{
        "type": "command",
        "command": "notify-send 'Claude Code' 'Needs attention'"
      }]
    }]
  }
}
```

On macOS, swap `notify-send` for `osascript -e 'display notification "…"'`; on WSL, use `powershell.exe` toast notifications.

<MarginNote variant="warning" label="Warning">
  Hooks fail silently by default. When one misbehaves, the usual cause is environment: hook commands run in a minimal shell, so a script that works in your terminal can fail on `PATH`. Use absolute paths, and add `>> /tmp/hook-debug.log 2>&1` while debugging. Then check it loaded with <code>/hooks</code>.
</MarginNote>

<Tip n="36" title="Non-negotiable → hook; advisory → CLAUDE.md.">
If a standard must hold every time regardless of what Claude is doing, make it a hook. If it's guidance Claude can reasonably adapt, put it in `CLAUDE.md`. Mismatching the two is the root of most "it didn't enforce my rule" surprises.
</Tip>

## MCP: consuming external services

The Model Context Protocol connects Claude to external services, databases, APIs, and custom tools through a standard protocol.<Citation src="anthropic-claude-code-mcp" /> This chapter is about **consuming** a server, not designing one. Adding a server is one command:

<BeforeAfter title="Raw API calls vs. a consumed MCP server">
<div slot="before">

```bash
# Manual: raw API calls to GitHub
curl -H "Authorization: token $GH_TOKEN" \
  https://api.github.com/repos/org/repo/issues
# Parse JSON, extract what you need, paste into Claude
```

</div>
<div slot="after">

```bash
# One command — Claude can now query GitHub directly
claude mcp add github --transport stdio -- npx @anthropic/mcp-github
claude mcp list          # Shows: github (stdio)
# Now in any session Claude can list/create issues + PRs,
# read repo contents, and search code across repos.
```

</div>
</BeforeAfter>

A consumed server provides **tools** (functions Claude can call), **resources** (data referenced via `@server:resource/path`), and prompts. Scopes mirror the rest of the config: **local** (this machine, default), **project** (`.mcp.json`, committed for the team), and **user** (`~/.claude/mcp.json`, personal global).

Servers run **locally over stdio** — your credentials never leave your machine; Claude talks to the server over stdio, not the network. Permission rules scope MCP tools with wildcards:

```json
{
  "permissions": {
    "allow": ["mcp__github__*"],
    "deny": ["mcp__github__delete_*"]
  }
}
```

<KeyIdea>
**The 10% context threshold.** When a server's tool definitions exceed ~10% of the context window, Claude switches to on-demand search instead of loading every tool's schema up front — keeping context efficient. The practical consequence: prefer a CLI for well-known services (it costs almost no context), and reserve MCP for systems that genuinely need typed, structured access mid-session. A single MCP server with dozens of verbose tool schemas can quietly eat a large slice of your window.
</KeyIdea>

Before installing any third-party server, run a quick provenance audit: check the README and manifest (a GitHub server shouldn't need filesystem write access), review the permissions it requests via `/permissions` and deny anything beyond its stated purpose, and test it first in a sandboxed session or throwaway worktree. Prefer servers from the official registry; for enterprise policy enforcement, use Managed MCP (covered in [**Chapter 16 — Enterprise Deployment**](/chapters/ch16-enterprise-deployment)).

## The worked composition

The framework pays off when a real problem decomposes into the _right_ rungs — not the heaviest ones.

<WorkedExample title="A security-review pass on every PR — walking the framework live">

**The need:** every time a teammate opens a PR, Claude should run a security-review pass and post inline comments on suspicious changes.

**Is it a reusable prompt?** Partly. The instructions ("look for SQL injection, leaked secrets, auth bypass; cite line numbers; suggest fixes") are reusable — but packaging them as `/security-review` relies on a human remembering to run it, and "remember to run the command" is exactly the step people forget. → a command is _part_ of the answer, not all of it.

**Is it a reusable procedure?** Yes: fetch the diff, scan the categories, produce structured findings, format a comment, post. That's a **skill** (`.claude/skills/security-review.md`) with the right tool access (`gh` for PR ops, `Read`/`Grep` for the scan).

**Does it need to fire automatically?** Yes — on PR open. But Claude Code hooks fire on _Claude's_ lifecycle events (`PreToolUse`, `Stop`, …), not GitHub's. The automation layer for a GitHub event is a **GitHub Action** that invokes the Claude CLI in headless mode. → a hook is _not_ the right tool here.

**Does it need external data?** Yes — the GitHub API. But the `gh` CLI already covers it. → **no new MCP server.** Consume `gh`.

**Final composition:** a **skill** that uses `gh` and the file-reading tools, invoked by the **CLI** in **headless mode** from a **GitHub Action** on PR open. Three rungs, each doing the lightest job that gets the result — and **zero custom MCP server built.**

<InsightBox title="Why this composition wins">
  The naive answer — "build an MCP server that integrates Claude with GitHub" — is a multi-week project that duplicates what `gh` already does. The framework answer — skill + CLI + Action — is a one-afternoon project on existing infrastructure. The lever was knowing which rung each _piece_ of the problem actually needs.
</InsightBox>

</WorkedExample>

## Plugins, IDEs, sandboxing, and CLI tools

The four-rung ladder covers most extension decisions. Four lighter surfaces round it out.

**Plugins** bundle skills, hooks, subagents, and MCP into a single installable unit — the fastest path from "nothing configured" to "productive."<Citation src="anthropic-claude-code-plugins" /> Browse the marketplace with `/plugin`; installed plugins land in `.claude/plugins/`. For typed languages, a **code-intelligence plugin** is worth installing immediately: it replaces text-based `Grep`/`Glob` navigation with precise "go to definition" / "find references" and automatic post-edit type checking — fewer file reads, smaller context, lower cost per message. Check the marketplace before building custom skills.

**IDE integrations** let Claude Code run in VS Code or JetBrains in addition to the terminal; all three share the same engine, `CLAUDE.md`, settings, and hooks.

- **VS Code** — a graphical chat panel, inline diffs with per-hunk accept/reject, `@`-mention file references (`@file.ts#5-10` for line ranges), plan mode as a commentable document, and conversation checkpoints. The extension bundles the CLI — no separate install — and works with Cursor too.<Citation src="anthropic-claude-code-vs-code" />
- **JetBrains** (IntelliJ, PyCharm, WebStorm, GoLand, and others) is **in beta**, so treat its surface as still settling. It's terminal-based — Claude runs in the IDE's integrated terminal with IDE features layered on (native diff viewer, current selection and open file shared as context, lint/type diagnostics forwarded to Claude) — and, unlike VS Code, it **requires the CLI installed separately**.<Citation src="anthropic-claude-code-jetbrains" />

Teams that standardize on one IDE can commit extension-specific settings; mixed-IDE teams should rely on `CLAUDE.md` and `settings.json`, which behave identically everywhere.

**Sandboxing** (`/sandbox`) gives OS-level isolation that restricts filesystem and network access, letting Claude run commands without per-action prompts because the sandbox prevents access beyond its boundaries — a safer alternative to `--dangerously-skip-permissions`.<Citation src="anthropic-claude-code-sandboxing" />

**CLI tools** are the most context-efficient way to reach an external service.<Citation src="anthropic-claude-code-best-practices" /> Install the CLI for services you use regularly — `gh` (GitHub), `aws`, `gcloud`, `sentry-cli` — and Claude invokes them via `Bash`. For an unfamiliar one: _"Use `tool --help` to learn the tool, then use it to solve X."_ A CLI consumes far less context than the equivalent MCP server, which is why it's the default for well-known services.

<InsightBox title="Where this chapter stops: consume vs. build">
  This is the **use-side** of extensions — which rung to reach for and how to consume one. The **design-side** lives in the **Architect's Reference**: designing MCP _servers_ (transport, tool schemas, resource models), authoring custom tools, and writing Agent SDK hooks programmatically. If the framework here lands you on "build a new MCP server" or "write a custom SDK hook handler," that volume is your next stop — not deeper into this chapter.
</InsightBox>

## Common pitfalls

<Pitfall title="Building an MCP server when consuming one would do">
The most expensive mistake in this chapter. MCP servers feel architecturally satisfying — typed tools, clean boundaries — and the urge to build "ours" over consuming "theirs" is real. Almost every external system worth talking to already has a CLI (`gh`, `aws`, `kubectl`, `psql`) Claude can invoke via `Bash` for a fraction of the cost. Map the existing landscape for a day before committing a week to building.
</Pitfall>

<Pitfall title="Reaching for hooks before commands or skills">
Hooks fire automatically and feel like the "real" engineering layer — which makes them the wrong default. A hook that runs `pytest` on every edit looks elegant but slows the loop, fires on irrelevant edits, and becomes an annoyance you eventually disable. Reach for a command first; promote to a skill if it grows branches; escalate to a hook only when _non-execution_ is unacceptable (blocking `rm -rf`, enforcing CI). Hooks are guardrails, not convenience automation.
</Pitfall>

<Pitfall title="Skipping the command stage entirely">
The most common failure isn't picking the wrong rung — it's leaping past commands straight to a skill or MCP server on day one. Most elaborate skills began as a prompt typed three times. If you're building a skill before you've felt the pain a command would have absorbed, you're paying maintenance cost you haven't earned yet.
</Pitfall>

<Pitfall title="Encoding a non-negotiable standard as a skill">
A skill is advisory — Claude can adapt or skip it, and someone has to invoke it. If "every commit passes the linter" lives in a skill, it holds only when Claude (or the human) remembers. Put enforceable standards in a `PreToolUse` hook; reserve skills for procedures that need judgment.
</Pitfall>

<Pitfall title="Letting a verbose MCP server eat the context window">
A single MCP server with dozens of richly-described tools can consume a large slice of context before you've typed a word — and past ~10% Claude falls back to on-demand tool search. Audit a server's tool count and description length, prune to the tools you use, and prefer a CLI when the service has a good one.
</Pitfall>

## Quick reference

The decision framework is the lever — not any single extension type. Reach for the lightest rung that does the job, and prefer consuming an existing extension over building one. Teams that always reach for MCP are building too much; teams that stop at commands when they can are winning.

| Mechanism | Reach for when | _Don't_ when | Lives at |
|---|---|---|---|
| **Command** | a repeatable prompt; same instructions used >2× | the work branches — use a skill | `.claude/commands/<name>.md` |
| **Skill** | a multi-step procedure; needs a tool envelope / "how we do X" | a single-step prompt (command) or cross-cutting enforcement (hook) | `.claude/skills/<name>/SKILL.md` |
| **Hook** | mechanical enforcement; can't trust the model alone | a `CLAUDE.md` instruction would hold | `settings.json` `hooks` |
| **MCP (consume)** | typed external access mid-session; an existing server covers it | a `Bash` call to a CLI (`gh`, `aws`, `psql`) suffices | `.mcp.json` |
| **MCP (build)** | every consuming option fails _and_ the integration is durable | almost always — see the Architect's Reference first | — |
| **Plugin** | packaging several of the above for distribution | single-extension cases — ship the file | `/plugin` → `.claude/plugins/` |
| **CLI / headless** | automation (CI, cron, GitHub Actions) | interactive work — use the TUI | the `claude` binary |
| **IDE** | in-editor surfacing your team shares | the team doesn't share an IDE | VS Code / JetBrains (beta) |

- **Promotion path:** prompt typed 3× → command → (needs branching) skill → (must run every time) hook → (needs external data) MCP.
- **Hooks:** ~29 lifecycle events; four handler types (`command` / `http` / `prompt` / `agent`); enforcement layer — always execute. Full reference: _Appendix C (forthcoming)_.
- **Skills:** `SKILL.md` per directory; precedence enterprise > personal > project; `context: fork` runs it as an isolated subagent; descriptions cost ~2% of context.
- **MCP / CLI:** servers run locally over stdio (credentials stay put); past ~10% context Claude switches to on-demand tool search — prefer a CLI for well-known services.

<Solution for="ch08-ex-1">
A skill is **advisory** — Claude can adapt or skip it, and a human (or Claude) has to remember to invoke it, so "every commit" isn't actually guaranteed. The standard is non-negotiable, which is the hook's job: a `PreToolUse` hook on `Bash(git commit*)` runs the linter whether or not anyone remembers, blocking the commit (exit code 2) on failure. The correct rung is a **hook**; the skill is one rung too low for an enforcement requirement.
</Solution>

<Solution for="ch08-ex-2">
`context: fork` and (optionally) `agent`. Setting `context: fork` runs the skill as a subagent in its **own context window**, so the file reads the audit generates don't consume your main session's context — only a summary returns when it finishes. `agent` selects which subagent type the fork uses (e.g. `general-purpose`). Together they let a context-hungry audit run in isolation; `model` can also pin it to `opus` for the heavier reasoning, but the isolation comes from the fork.
</Solution>

<Solution for="ch08-ex-3">
`gh` is invoked via `Bash` and costs almost no context, whereas an MCP server loads its tool schemas into the window — and a verbose server can pass the ~10% threshold where Claude falls back to on-demand tool search, paying context for capability you may not need for a one-shot read. So for well-known services with a good CLI, the CLI is the right default. The MCP server wins when you need **typed, structured access mid-session** — repeated, schema-validated calls where parsing raw CLI output would be brittle, or where Claude needs the server's resources (`@server:resource/path`) rather than ad-hoc command output.
</Solution>
