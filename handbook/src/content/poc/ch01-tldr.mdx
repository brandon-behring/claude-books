<KeyIdea>
Claude Code is an agent, not a chatbot. It runs a tool-call loop with finite context. Treat it like a system (clear inputs, verification, deliberate context control) and you get system-quality output. Treat it like a chatbot and you get chatbot quality.
</KeyIdea>

## What Claude Code actually is

**An agent loop**: read prompt → reason → pick a tool → observe result → decide next step → repeat. Three properties shape what the loop can do:

- **Context window** — finite. Every tool call costs tokens. When it fills, old info fades.
- **Tool use** — Claude only acts through tools (`Read`, `Edit`, `Write`, `Bash`, `Glob`, `Grep`). No tool = no action.
- **Configuration layers** — four layers (CLAUDE.md, rules, settings, output styles) shape behavior.

## Four engineering principles

| # | Principle | One-line application |
|---|---|---|
| 1 | **Never fail silently** | Errors reported with message, root cause, and 2-3 resolution options |
| 2 | **Simplicity over complexity** | 20-line function with clear name > 5-line function with 3 abstractions |
| 3 | **Immutability by default** | Return new data; mark mutations explicitly (`update_` prefix) |
| 4 | **Fail fast with diagnostics** | Stop on problems with full context (what failed, what was expected, what to do) |

These principles **apply to all code**, not just AI-generated. They matter *more* when AI is involved because AI amplifies both good and bad patterns in the codebase it reads.

## The codebase is the curriculum

<MarginNote variant="note" label="Key">
  Configuration reinforces; it does not replace.
</MarginNote>

Claude learns patterns from the code it reads. **Ensure the code Claude reads exemplifies the patterns you want Claude to produce.** Investment-wise: the single most valuable thing you can do before pointing Claude at an existing project is to clean up the patterns it will encounter most often (entry points, frequently-imported utilities, data loaders).
