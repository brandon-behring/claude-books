<KeyIdea>
A session is a **loop**, not a prompt: prompt → observe → refine → commit. Make Claude precise with three basics — be specific about *what*, structure multi-part tasks, and include a **verification criterion**. Keep context clean (`/clear`, `/compact`, the two-failure rule), **interrupt early** when Claude drifts, and **plan before risky changes**.
</KeyIdea>

## The session loop

```text
PROMPT → OBSERVE → REFINE → COMMIT   (refine loops back; a wrong turn /clears)
```

**Observe is not optional** — read the diff, run the tests. One logical change per commit. The two steps that decide quality are **prompt** and **refine**.

## Three prompting basics

| Basic | What it does |
|---|---|
| **Be specific about *what*, not *how*** | Names the symptom + location + goal, so Claude stops guessing |
| **Think step by step** | An explicit structure for multi-part tasks; fewer one-pass regressions |
| **Include verification criteria** | The success bar in the prompt — Claude self-corrects before you see it |

None is about writing code. They are about defining "correct" precisely enough that Claude can produce it — your value shifts from *implementation* to *specification*.

## Context hygiene

- **`/clear`** — wipe and start fresh, *between unrelated tasks* (`CLAUDE.md` reloads).
- **`/compact`** — summarize to *continue* a long task whose context is getting large.
- **Two-failure rule** — after two failed corrections, stop. `/clear` and write the prompt you wish you had written first. A third round fights a cluttered desk.

## Course-correcting

Durable idea: **interrupt early, recover cheaply.** Lightest to heaviest:

```text
Interrupt (Esc)  → stop mid-action, context kept     (wrong RIGHT NOW)
Rewind (/rewind) → restore code/conversation to a checkpoint
"Undo that"      → revert the last change in words
/clear           → reset entirely                    (whole direction wrong)
```

## Plan Mode

A **read-only** permission mode — Claude reads and plans but cannot edit. Use it for changes you *cannot* describe as a one-sentence diff (multi-file, unfamiliar code, hard to reverse). Four phases: **explore → plan → implement → commit**. The payoff: a misunderstanding surfaces in a plan you can read, not in code you must unwind. Skip it for typos and renames.
