## The chapter in one paragraph

Claude Code is an agent, not a chatbot. It reads your prompt, picks a tool, observes the result, and decides what to do next — a loop that runs until the task completes or you intervene. Three properties — **a finite context window**, **tool use** as the only way to act, and **configuration layers** that shape behavior — determine what Claude can do and how well. Four principles — **never fail silently, simplicity over complexity, immutability by default, fail fast with diagnostics** — apply to every recommendation that follows. The codebase Claude reads becomes the curriculum it learns from.

<KeyIdea>
The same tool produces dramatically different results depending on which mental model the developer holds. Treating Claude as a chatbot — vague prompts, no verification, no context management — produces chatbot-quality output. Treating Claude as a system with finite context and observable tool calls produces system-quality output.
</KeyIdea>

## What Claude Code actually is

<MarginNote variant="note" label="Official">
  Claude Code is an agentic coding tool that operates directly in your terminal. <a href="https://code.claude.com/docs/en/overview">code.claude.com/docs/en/overview</a>
</MarginNote>

Claude Code runs an **agent loop**: it reads your prompt, reasons about the task, selects a tool (read a file, edit code, run a command), observes the result, and decides what to do next. This cycle repeats until the task is complete or Claude asks for clarification.

Three properties define the system:

- **Context window** — Everything Claude knows about your task (your prompt, files it has read, tool results, conversation history) occupies a finite window. When the window fills, older information fades. Context is not infinite memory; it is a workspace.
- **Tool use** — Claude does not type code into a file. It calls tools: `Read`, `Edit`, `Write`, `Bash`, `Glob`, `Grep`. Each tool call consumes context (the request and the result) and produces observable effects. Every tool call has a cost in context tokens.
- **Configuration layers** — Claude's behavior is shaped by four layers: `CLAUDE.md` (project knowledge), rules (conditional context), settings (permissions and behavior), and output styles (communication patterns). These layers are not suggestions — they are the operating system through which Claude interprets your project.

<InsightBox title="Why this mental model matters">
  Developers who treat Claude as a chatbot prompt it like one: vague requests, no verification, no context management. Developers who understand the agent loop prompt it like a system: clear inputs, verification criteria, deliberate context control. The same tool produces dramatically different results depending on which mental model the developer holds.
</InsightBox>

<WorkedExample id="agent-loop-walkthrough" title="The agent loop in one session — &quot;find the auth function&quot;">

Annotated step by step so you can see the loop:

```
You: "Find the function that handles user authentication."

Claude reasons: "I need to search the codebase for auth-related symbols."
Claude calls Grep:        ← tool call (costs ~50 tokens of context)
  pattern: "auth"
  output: 23 matches across 8 files

Claude reasons: "Too many. Let me narrow."
Claude calls Grep:        ← tool call (costs ~50 tokens of context)
  pattern: "def authenticate" OR "def login"
  output: 2 matches in src/auth.py

Claude calls Read:        ← tool call (costs ~400 tokens of context — the file)
  file: src/auth.py
  output: full file content

Claude responds: "src/auth.py line 47 is `authenticate_user(...)`.
  It validates credentials against the users table and issues a JWT."
```

Each `← tool call` is the loop iterating. The conversation includes everything from your prompt through Claude's response — that whole transcript stays in context until you `/clear` or `/compact`. **A 30-tool-call session can spend half its context budget on tool outputs alone.** This is why "context as currency" (Ch 5) becomes a load-bearing concept later in the handbook.

</WorkedExample>

## Four engineering principles

These four principles underlie every recommendation in this handbook. They apply to code written by AI, code written by humans, and code written by both together.

### 1. Never fail silently

Every error must be explicitly reported with recovery options. Silent failures are the most expensive kind of bug: they produce incorrect results that look correct, propagate through downstream systems undetected, and surface only when the damage is difficult to reverse.

<BeforeAfter>
<div slot="before">

```markdown
# CLAUDE.md
Handle errors gracefully.
```

</div>
<div slot="after">

```markdown
# CLAUDE.md
If you encounter an error, report it immediately
with the error message, your analysis of the root
cause, and 2-3 options for resolution.
Never work around errors silently.
```

</div>
</BeforeAfter>

<MarginNote variant="tip">
  The "after" version is longer but produces better outcomes. Precision in error handling instructions compounds across every session.
</MarginNote>

In AI-assisted development, this has an additional dimension: Claude must be configured to surface problems rather than work around them. A vague "handle errors" invites Claude to catch and suppress. An explicit "report errors with analysis" makes every failure visible and actionable.

### 2. Simplicity over complexity

Short functions. Flat structure. Self-documenting code. A 20-line function with a clear name is superior to a 5-line function with three levels of abstraction.

<InsightBox title="Why simplicity matters more with AI">
  Simple code is easier for Claude to reason about, test, and modify correctly. Complex abstractions increase the probability of subtle bugs in AI-generated code because each layer of indirection is another opportunity for misunderstanding. When Claude reads your codebase to learn conventions, simple patterns propagate accurately; complex patterns propagate errors.
</InsightBox>

### 3. Immutability by default

Return new data structures; mark mutations explicitly. Pure functions are easier to test, easier to parallelize, and easier for Claude to reason about.

When mutation is necessary (performance, I/O, state management), make it explicit: name the function `update_`, use mutable types deliberately, and document the side effects. The reader — human or AI — should never be surprised by what a function modifies.

### 4. Fail fast with diagnostics

Stop immediately on problems with full context: what failed, what was expected, what was received, and what the caller can do about it.

```python
def process_data(df: pd.DataFrame, min_rows: int) -> pd.DataFrame:
    if len(df) < min_rows:
        raise ValueError(
            f"Need {min_rows} rows, got {len(df)}. "
            f"Check data source or reduce min_rows parameter."
        )
    # ... process
```

<MarginNote variant="tip">
  Error messages that include the violation AND a suggested fix reduce debugging time dramatically.
</MarginNote>

The error message includes: what went wrong (`Need N, got M`), where to look (`data source`), and what to do about it (`reduce min_rows`). The difference between a 10-second fix and a 30-minute investigation.

## The codebase is the curriculum

These principles are not Claude-specific — they are engineering principles that become *more* important when AI is generating code. AI amplifies both good and bad patterns.

If your codebase follows these principles, Claude's contributions will follow them too. If your codebase is silent about errors, full of mutable state, and deeply abstracted, Claude's contributions will be the same.

The single most valuable thing you can do before using Claude on an existing project is to ensure that the code Claude will read **exemplifies the patterns you want Claude to produce**. Configuration (covered in [Ch 2](#) and revisited in [Ch 9](#)) reinforces this, but the codebase itself is the primary teacher.

## Pitfalls (common first-session mistakes)

<Pitfall title="Treating Claude as a chatbot">
Vague requests, no verification, no context management. Reads like the "before" of every example in this chapter. The fix is the rest of this chapter — but specifically: write verification criteria in your prompt, ask Claude to read before it edits, and `/clear` between unrelated tasks.
</Pitfall>

<Pitfall title="Ignoring tool-call cost">
Letting Claude read 30 files when 3 would do. Burns context for no signal. Always start by naming the files or directories Claude should look at, even when you don't know the exact path — Claude is better at scoping a `Grep` to a directory than at deciding which of 100 files to read first.
</Pitfall>

<Pitfall title="Skipping CLAUDE.md">
Relying on prompt-only configuration. CLAUDE.md is the **persistent** layer that survives `/clear`; prompts are session-only. The two-minute investment in a 60-line CLAUDE.md pays back every session, forever.
</Pitfall>
