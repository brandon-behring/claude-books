> **Use this when**: you need to look up *one specific thing* about extending Claude — which file path a skill lives at, which hook event fires when, whether to consume vs build an MCP server. Not a tutorial; not a TL;DR. Look-up oriented.

## Decision flowchart

```
┌─────────────────────────────────────────────────────────────────┐
│ START: what do you need?                                        │
│   │                                                             │
│   ▼                                                             │
│ Reusable prompt only?                                           │
│   ├── YES ─► .claude/commands/<name>.md          (cheapest)     │
│   └── NO ▼                                                      │
│ Reusable procedure (multi-step, own tool envelope)?             │
│   ├── YES ─► .claude/skills/<name>.md                           │
│   └── NO ▼                                                      │
│ Must fire on Claude lifecycle event (PreToolUse, Stop, etc.)?   │
│   ├── YES ─► .claude/settings.json  "hooks": {...}              │
│   └── NO ▼                                                      │
│ Need external service / typed tools during the session?         │
│   ├── YES ─► Does an existing MCP server cover it?              │
│   │           ├── YES ─► add to .mcp.json (CONSUME)             │
│   │           └── NO  ─► Architect's Ref before building        │
│   └── NO ▼                                                      │
│ Packaging multiple of the above for distribution?               │
│   ├── YES ─► Plugin                                             │
│   └── NO  ─► Reconsider — you may not need an extension         │
└─────────────────────────────────────────────────────────────────┘
```

**Stop at the first YES.** Stopping early is the goal.

## Extension type × file location × scope × invocation

| Type | Project path | User path | Scope | Invocation |
|---|---|---|---|---|
| **Slash command** | `.claude/commands/<name>.md` | `~/.claude/commands/<name>.md` | Per-repo or global | `/name [args]` |
| **Agent skill** | `.claude/skills/<name>.md` | `~/.claude/skills/<name>.md` | Per-repo or global | `/name [args]` or NL match on `description` |
| **Hook** | `.claude/settings.json` `hooks` field | `~/.claude/settings.json` `hooks` field | Per-repo or global | Auto, on lifecycle event |
| **MCP server** | `.mcp.json` | `~/.mcp.json` | Per-repo or global | Auto-loaded at session start |
| **Plugin** | Bundled module | Bundled module | Distributable | Per the plugin's manifest |
| **CLI** | `claude` binary | `claude` binary | System-wide | `claude -p "..."` (headless), etc. |
| **IDE integration** | Extension marketplace | Extension marketplace | Per-IDE | Editor-specific commands |

<MarginNote variant="tip" label="Tip">
  Project scope (<code>.claude/</code>) lives in the repo and version-controls with the team. User scope (<code>~/.claude/</code>) is personal and applies across all your repos.
</MarginNote>

## Hook event quick reference

<MarginNote variant="warning" label="Warning">
  The hook event list grows over time; this is a snapshot at 2026-05-23. Check <code>docs.claude.com</code> for additions.
</MarginNote>

| Event | Fires when | Common use |
|---|---|---|
| `PreToolUse` | Before any tool call | Block dangerous commands; gate writes |
| `PostToolUse` | After a tool call returns | Auto-format on Edit; log tool I/O |
| `UserPromptSubmit` | When you submit a prompt | Inject context; redact secrets |
| `Stop` | When Claude finishes a turn | Run tests; post status to a channel |
| `SubagentStop` | When a sub-agent completes | Aggregate sub-agent results |
| `Notification` | On user-facing notification | Forward to system notifications |
| `SessionStart` | New session begins | Load extra context; announce session |
| `SessionEnd` | Session ends | Persist state; archive transcript |

**Shape**: each hook is `{ matcher: "...", command: "shell command" }`. The shell command's stdout/stderr can modify, block, or annotate the event. Exit code 0 = allow; non-zero = block.

## Consume vs build (MCP)

| Question | Consume | Build |
|---|---|---|
| Does an existing MCP server cover this? | YES → consume | NO ↓ |
| Does a CLI (`gh`, `aws`, `psql`, etc.) cover it via `Bash`? | YES → use CLI | NO ↓ |
| Is the integration *durable* (you'll use it for years)? | NO → use CLI | YES ↓ |
| Do you have engineering capacity for a real software project? | NO → defer | YES → build (see Architect's Reference) |

**Default**: consume. Building an MCP server is a real software project — protocol, transport, tool schemas, error handling, deployment, security review. Don't take it on unless the consuming options have demonstrably failed.

## Skill frontmatter shape

```markdown
---
name: <kebab-case-name>           # how it's invoked: /kebab-case-name
description: <one sentence>       # what Claude matches against
# optional:
# allowed-tools: [Read, Edit, Bash]
---

# Skill body
... procedure here ...
```

## Slash command shape

```markdown
# .claude/commands/<name>.md
... the prompt body Claude reads when /name is invoked ...
# Args, if any, are interpolated as {{1}}, {{2}}, ...
```

## Hook config shape

```jsonc
// .claude/settings.json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "command": ".claude/hooks/deny-dangerous-bash.sh"
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "command": ".claude/hooks/format-on-write.sh"
      }
    ]
  }
}
```

## Common compositions (recipes)

| Use case | Composition |
|---|---|
| "Every PR gets a security-review pass" | Skill + CLI headless + GitHub Action |
| "Block dangerous bash commands" | `PreToolUse` hook with a deny-list script |
| "Auto-format on save" | `PostToolUse` hook matching `Edit\|Write` |
| "Run the release procedure consistently" | Skill (`.claude/skills/release-checklist.md`) |
| "Translate every error message" | Slash command (`/explain-error`) |
| "Typed access to our Postgres" | Consume an existing Postgres MCP server |
| "Headless CI summary on every push" | CLI in headless mode invoked from CI |

## Project vs user scope decision

| Need | Scope |
|---|---|
| Team needs to share this | Project (`.claude/`) — version controls with the repo |
| Only you use it across all your repos | User (`~/.claude/`) |
| Both — team needs it + you want it elsewhere | Project + symlink to user, OR duplicate (file is tiny) |
