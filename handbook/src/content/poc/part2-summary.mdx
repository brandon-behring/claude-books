## Chapter summaries

### Ch 5 — Context as Currency

**Core claim**: every tool call costs tokens, and a session's working memory degrades as it fills. `/clear`, `/compact`, `/rewind`, `/btw` are the four moves; the two-failure rule is the trigger that gets you to use them. Session naming + `CURRENT_WORK.md` extend useful sessions across days.

- [TL;DR](/poc/ch05-tldr) — 4-command table + the two-failure rule (~5 min)
- [How-to: recover from runaway context](/poc/ch05-howto-recover-runaway-context) — concrete recovery, 7 steps (~5 min)
- [Cheat sheet](/poc/ch05-cheat-sheet) — context-budget ASCII diagram + decision matrices + the "do nothing" row

### Ch 6 — Thinking Together

**Core claim**: delegating to Claude as an oracle produces sycophantic mush. Collaborating with Claude as a thinking partner — via hypothesis debugging, pre-mortem, Feynman, and the interview pattern — surfaces real reasoning. The interview pattern is the third frame between "delegate" and "collaborate" and the one most teams underuse. ADRs preserve the decisions that come out of these sessions.

- [Tutorial](/poc/ch06-tutorial) — narrative with the "Claude starts wrong; reader notices the sycophancy; interview pattern recovers" arc through the 4 named patterns (~15 min)
- [TL;DR](/poc/ch06-tldr) — the interview-as-third-frame insight + pattern table (~5 min)
- [How-to: run a pre-mortem](/poc/ch06-howto-pre-mortem) — 7-step procedural with the actual prompts (~5 min)
- [Cheat sheet](/poc/ch06-cheat-sheet) — copy-paste prompt templates for all 4 patterns + sycophancy red flags + ADR template

### Ch 7 — Testing & Verification

**Core claim**: the difference between "Claude said it works" and "it actually works" is verification. Six layers (type-check / lint / unit / integration / e2e / manual) plus a 7th (domain correctness — only humans can do it). Phase-appropriate standards: prototype skips layers 4-7; production runs all 7. Verification criteria belong in `CLAUDE.md`; enforcement belongs in hooks.

- [Tutorial](/poc/ch07-tutorial) — narrative with the "all 6 automated layers pass and the code is still wrong" worked example (~15 min)
- [TL;DR](/poc/ch07-tldr) — the 6+1 layer reference table + phase shortlist (~5 min)
- [How-to: add a verification gate](/poc/ch07-howto-add-verification-gate) — `CLAUDE.md` block + `Stop` hook + `verify.sh` (~5 min)
- [Cheat sheet](/poc/ch07-cheat-sheet) — ASCII pyramid + phase × layer matrix + hook event reference + per-phase verification-criteria templates

### Ch 8 — Extending Claude

**Core claim**: there is a decision framework before there is any extension type. Reach for the lightest extension that does the job; build new only when consuming fails. Order of escalation: custom slash command → skill → hook → MCP server. Hooks are for enforcement, not feature-add. MCP is for genuine third-party integration, not as a default. Designing MCP servers and Agent SDK hooks lives in the Architect's Reference; this chapter teaches consumption.

- [Tutorial](/poc/ch08-tutorial) — narrative walking through "security review on every PR" live through the decision: skill, CLI, GitHub Action — zero new MCP server (~15 min)
- [TL;DR](/poc/ch08-tldr) — decision-framework punchline + extension type × when-to-reach table (~5 min)
- [How-to: write your first skill](/poc/ch08-howto-first-skill) — end-to-end build of `.claude/skills/release-checklist.md` (~5 min)
- [Cheat sheet](/poc/ch08-cheat-sheet) — ASCII decision flowchart + extension-type matrix + hook event reference + consume-vs-build heuristic

## What you'll have built by end of Part II

- A **session-management vocabulary** (`/clear` / `/compact` / `/rewind` / `/btw`) you reach for automatically when the two-failure rule fires
- A **`CURRENT_WORK.md`** that survives across sessions for at least one ongoing piece of work
- A **collaboration repertoire** (hypothesis debugging, pre-mortem, Feynman, interview) — at least one of each invoked in a real situation
- A **`CLAUDE.md`** that names verification criteria (no "handle errors gracefully" mush)
- At least one **hook** in `.claude/settings.json` that enforces something you care about (a `Stop` hook that runs `verify.sh`, or a `PreToolUse` that blocks `rm -rf`)
- At least one **custom skill** in `.claude/skills/` that packages a recurring procedure
- At least one **ADR** in `.claude/decisions/` capturing a non-obvious choice

That's enough to make Claude reliable across the long tail of work, not just per-session. Parts III–IV build on this foundation: hub-and-spoke configuration at scale, subagents for parallelism, team governance, enterprise deployment.
