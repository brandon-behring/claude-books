## The four patterns at a glance

| Pattern | Stage of task | Goal |
|---|---|---|
| **Hypothesis debugging** | A symptom appears | Enumerate causes, rank them, identify cheapest disambiguating evidence |
| **Pre-mortem** | Before irreversible decision | Surface concrete failure modes and detection signals |
| **Feynman** | Self-check on understanding | Find vague/undefended claims in your own explanation |
| **Interview** | Start of any non-trivial task | Get Claude to ask questions whose answers change the build |

## Copy-paste prompt templates

### Hypothesis debugging

```
Symptom: <paste error, log, stack trace, or anomaly>

Don't fix this yet. List 3 hypotheses for the root cause,
ranked by your confidence. For each, name the single piece
of evidence that would confirm or rule it out. Identify the
cheapest evidence to gather first.
```

### Pre-mortem

```
I am about to commit to <one-sentence decision>.

Imagine it is six months from now. We adopted this decision
and it failed badly. Write the post-mortem.

Requirements:
  - Be specific. Name the actual failure mode, not "scaling
    problems."
  - Identify the signals we should have noticed at decision
    time.
  - Identify the question we should have asked.
  - Cover at least 3 failure modes, ranked by likelihood.

Be harsh. The point is to find what we're missing.
```

### Feynman

```
I'm going to explain how <topic> works in my own words. Read
it as if you were a smart undergrad encountering this for the
first time. Then identify:

  1. Sentences that are vague or use jargon I haven't defined
  2. Claims I'm asserting without justification
  3. Places where you'd ask me a clarifying question

<your explanation here>
```

### Interview (start-of-task)

```
Before you start: interview me. Ask 5-7 questions whose
answers would substantially change what you build. Number
them so I can answer in batch. Don't propose a solution
until I've answered.
```

### Interview (mid-task, recover from sycophancy)

```
Before applying anything, interview me about this. What
would you need to see in the codebase to be confident in
your diagnosis? List 2-3 alternative hypotheses with the
evidence that would distinguish them.
```

## Anti-sycophancy red flags

<MarginNote variant="warning" label="Warning">
  All three flags are highest at session-start.
</MarginNote>

| Red flag | What it means | Rewording fix |
|---|---|---|
| **"You're absolutely right"** in response to a claim you haven't verified | Claude is agreeing with your framing, not the evidence | Strip the load-bearing claim from your prompt; ask Claude to reach it from evidence |
| **A fix proposed before any `Read` tool call** | Pattern match, not diagnosis | "Before proposing a fix, read `[relevant file]` and tell me what you see" |
| **Exactly one diagnosis with no alternatives** | Premature closure | "List 2-3 alternative hypotheses ranked by confidence" |

## Anti-sycophancy prompt rewordings

| Sycophantic prompt | Anti-sycophantic rewording |
|---|---|
| "The bug is in the cache layer — fix it" | "Something is producing stale reads in this endpoint. List the components in the request path and rank which is most likely the source." |
| "We should use Postgres for this — agree?" | "We're choosing a datastore for `[use case]`. List 3 candidates, then for each name the biggest reason it would be the wrong choice here." |
| "Is this a good design?" | "Critique this design. What are the 3 most likely failure modes in production? What would a reviewer skeptical of it ask first?" |
| "Was that the right fix?" | "Walk through what that fix actually changes and what could still be broken. What test would catch a regression of the original symptom?" |

## ADR template (3-section)

```markdown
# ADR NNNN: <Short decision title>

Date: YYYY-MM-DD
Status: Proposed | Accepted | Superseded by NNNN

## Context

- What was true at decision time
- What alternatives existed (name 2-3, with one-line rejection reasons)
- What constraints applied (technical, organizational, time)
- ~5 bullets

## Decision

We will <one-sentence statement of what was chosen>.

We chose this over <named alternative> because <1-2 reasons>.

## Consequences

What becomes easier:
- <bullet>
- <bullet>

What becomes harder:
- <bullet>
- <bullet>

Signals that would tell us we chose wrong (from the pre-mortem):
- <specific, measurable signal — e.g. "query p99 > 200ms"> → <action>
- <specific, measurable signal> → <action>

Mitigations in place at commit time:
- <cheapest mitigation from the pre-mortem>
```

<MarginNote variant="tip" label="Tip">
  The "Signals that would tell us we chose wrong" bullets are the load-bearing part. They turn the ADR from a justification into a falsifiable record.
</MarginNote>

## Quick reference: which pattern when

| Situation | Pattern |
|---|---|
| Test fails and you don't know why | Hypothesis debugging |
| Production metric drifts | Hypothesis debugging |
| About to pick a database / vendor / framework | Pre-mortem |
| About to deprecate or restructure | Pre-mortem |
| About to ship a feature you'll be supporting for 3+ years | Pre-mortem |
| You're about to teach this to a junior | Feynman |
| You're writing docs and the prose feels mushy | Feynman |
| Starting any task with > 1 hour of implementation work | Interview |
| Claude just produced something fluent that you can't evaluate | Interview (mid-task) |

## Session hygiene

- Use **interview** at session-start; sycophancy is highest when context is lowest
- After **hypothesis debugging**, name the file you read and the hypothesis it disambiguated in the next prompt — that grounds Claude in evidence
- After a **pre-mortem**, write the ADR in the same session while reasoning is fresh
- After **Feynman**, the deliverable is *your edited explanation*, not Claude's summary
