> **You have**: a Claude Code session that is 25-40 messages deep, has read 8-15 files, and is showing signs of degradation (repeated suggestions, forgotten constraints, contradictions). The task is **mid-stride** — partially completed work you do not want to lose. **You want**: a clean, fast context with the productive state preserved.

## Prerequisites

- Claude Code installed and an active session that is showing degradation.
- Write access to the current working directory (for `CURRENT_WORK.md`).
- A real task in progress — this how-to does not apply to chat-only sessions.
- 10 minutes.

## Steps

### 1. Confirm degradation with the two-failure check

Before resetting anything, confirm the session actually needs intervention. Look for one of:

- Claude proposed the same fix you already discarded.
- Claude asked you something you already told it earlier in the session.
- Claude contradicted a constraint you established 10+ messages ago.

<MarginNote variant="warning" label="Warning">
  One repeat suggestion is data. Two on the same problem is a verdict. Do not reset on a single misfire.
</MarginNote>

If you see two of these signals (or one of them twice), proceed. If not, the session is probably fine — a sharper prompt may resolve it.

### 2. Capture the state to `CURRENT_WORK.md`

Before any reset, write down where you are. Open or create `CURRENT_WORK.md` at the repo root:

```markdown
# Current work — <session-name>

## What I'm doing
Fixing the session-drop bug in auth middleware
(sessions die after exactly 5 min).

## What's confirmed (do not re-explore)
- Not the JWT — token is valid at drop time
- Not cookie SameSite — same behavior in incognito
- Test: `pytest tests/auth/test_session.py::test_5min_persist`

## Suspect
Redis TTL on the session cache.
`SessionStore.get` in `src/auth/session.py:47`.

## Next step
Instrument the Redis client to log SETEX calls
during the test run.
```

Three to five short sections is enough. The point is to capture the **structural state** Claude can re-derive faster than you can re-explain it.

### 3. Decide between `/compact`, `/rewind`, and `/clear`

Pick the cheapest tool that solves your specific situation:

| You observe | Use | Why |
|---|---|---|
| Context is bloated but the recent work is good | `/compact` | Preserves the conversation thread; frees old tool outputs |
| Last 3-5 messages went off the rails; earlier work is good | `/rewind` | Surgical undo without losing earlier progress |
| Two failures on the same problem; whole session feels contaminated | `/clear` + reload `CURRENT_WORK.md` | Clean slate, fresh attention budget |

<MarginNote variant="tip" label="Tip">
  When in doubt, `/compact` first. If quality does not return within 2-3 messages, escalate to `/clear`.
</MarginNote>

### 4. Execute the chosen reset

For `/compact` — accept any summarization prompt Claude generates:

```
/compact

[Claude summarizes the conversation. Confirm or refine the summary.]
```

For `/rewind` — specify how far back if your client supports it:

```
/rewind 5

[Backs the session out 5 turns. Earlier history intact.]
```

For `/clear` — wipe and start clean:

```
/clear

[Session is empty except CLAUDE.md and auto-memory.]
```

### 5. Re-prime the session with the lesson encoded

After a `/clear` (or even after `/compact`), the first new prompt should:

1. Point Claude at `CURRENT_WORK.md`.
2. **Encode the lesson** from the failed attempts so Claude does not repeat them.

Example first prompt after `/clear`:

```
Read CURRENT_WORK.md.

We previously tried (and ruled out):
- Increasing the cookie max_age
- Changing JWT expiration
- Forcing a session refresh on every request

Do NOT propose these again. Start from the
Redis TTL hypothesis described in the file.
```

<KeyIdea>
The "lesson encoded" line is what makes the reset productive instead of wasteful. Without it, the fresh session will likely walk the same path you just walked. The two-failure rule fixes the budget; the encoded lesson fixes the trajectory.
</KeyIdea>

### 6. Verify quality has returned

In the next 2-3 messages, watch for:

- Claude proposes a **new** approach (not one you ruled out).
- Claude acknowledges the constraints from `CURRENT_WORK.md` and the lesson note.
- Tool calls are targeted (it goes straight to `src/auth/session.py:47`, does not re-read 12 files to orient).

If yes, the reset worked. If you still see repeat suggestions or confusion, you likely needed `/clear` instead of `/compact` — escalate now rather than later.

## Expected outcome

A clean session with the same productive context you had before, minus the bloat. You should be able to resume the actual task within 1-2 prompts, with Claude operating at the quality level you had at message 5 of the original session.

## When this fails

- **`/compact` was not enough** — the summary preserved the bad pattern. Escalate to `/clear` and re-prime from `CURRENT_WORK.md`.
- **`CURRENT_WORK.md` was too vague** — Claude re-explores territory you already ruled out. Add a "Confirmed dead ends" section with explicit "do not retry" instructions.
- **The new session repeats the same failure** — the bug itself may be subtler than you think; the session is not the problem. Step away from Claude for 10 minutes and re-read the failing test yourself; you may have been pattern-matching to the wrong category of bug.
