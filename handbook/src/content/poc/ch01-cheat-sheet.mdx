## Agent loop

```
┌─────────────────────────────────────────────────────────┐
│  prompt                                                 │
│    │                                                    │
│    ▼                                                    │
│  reason ─────────► no tool needed? ──► end_turn ─► you   │
│    │                                                    │
│    │ tool needed                                        │
│    ▼                                                    │
│  pick tool ──► call tool ──► observe result             │
│                                  │                      │
│                                  ▼                      │
│                              decide:                    │
│                              - more tools needed? loop  │
│                              - done? end_turn           │
└─────────────────────────────────────────────────────────┘
```

Each iteration costs context tokens (request + response).

## Three system properties

| Property | What it means | Practical consequence |
|---|---|---|
| **Context window** | Finite workspace; everything (prompt, files read, tool results, conversation history) lives here | Tool calls have cost — every `Read`, `Bash`, `Grep` consumes tokens |
| **Tool use** | Claude only acts through tools; cannot type into files directly | Observe each tool call; pattern-match what's happening |
| **Configuration layers** | Four layers shape behavior (CLAUDE.md / rules / settings / output styles) | Permanent state survives `/clear`; prompts don't |

## Four engineering principles

| # | Principle | Trigger | Anti-pattern | Right pattern |
|---|---|---|---|---|
| 1 | **Never fail silently** | Function catches an error | `except: pass` | `raise` with context, or log + return Result type |
| 2 | **Simplicity over complexity** | Tempted to add abstraction | 5-line function with 3 layers of indirection | 20-line function with one clear job + descriptive name |
| 3 | **Immutability by default** | Function modifies its input | `def update(df): df['x'] = ...` (mutates) | `def with_x(df): return df.assign(x=...)` (returns new) |
| 4 | **Fail fast with diagnostics** | Precondition not met | `assert len(df) > 0` (no message) | `raise ValueError(f"Need rows, got 0. Check {source}.")` |

## Configuration layer hierarchy

Loaded in this order (later loads override earlier on conflict):

```
1. ~/.claude/CLAUDE.md            (your global preferences)
2. <repo-root>/CLAUDE.md          (workspace conventions)
3. <repo-root>/.claude/rules/*.md (path-scoped conditional rules)
4. <package>/CLAUDE.md            (package-specific overrides)
5. <subdir>/CLAUDE.md             (most-specific overrides — wins)
```

**Specific wins.** `<subdir>/CLAUDE.md` overrides `<repo-root>/CLAUDE.md` on conflict.

## CLAUDE.md anti-patterns + fixes

| Anti-pattern | Fix |
|---|---|
| "Handle errors gracefully." | "If you encounter an error, report it immediately with the error message, your analysis of the root cause, and 2-3 options for resolution. Never work around errors silently." |
| "Write tests." | "Every route handler gets a `.spec.ts` next to it. Run `pnpm test <file>` to verify before considering the change complete." |
| "Follow the existing patterns." | "Check `src/routes/auth/login.spec.ts` as the canonical test pattern. Check `db/queries/users.ts` as the canonical query pattern. Match the structure exactly." |
| 200-line CLAUDE.md with stack details, install steps, history | Trim to ≤60 lines. Move install steps to README. Move history to git log. CLAUDE.md is conventions Claude needs *every session*, not onboarding. |

## Tool reference (when each is right)

| Tool | When to reach for it | When NOT to use |
|---|---|---|
| `Read` | Need to see file content for context | When file is huge — use `Grep` to scope first |
| `Grep` | Searching for patterns / symbols across files | When you want to discover file structure — use `Glob` |
| `Glob` | Listing files matching a pattern | When you need to read content — use `Grep` or `Read` |
| `Edit` | Targeted change to part of a file | When the change is most of the file — use `Write` |
| `Write` | Creating a new file or rewriting most of one | When you only need to change a few lines — use `Edit` |
| `Bash` | Running tests, commands, scripts | For file ops — use the dedicated file tools |

## Session-start checklist

Run through this before any complex task:

- [ ] Is there a CLAUDE.md at the repo root?
- [ ] Does it cover the conventions for *this* specific task?
- [ ] Are deny rules in `settings.json` for anything destructive (`git push --force`, `rm -rf`, etc.)?
- [ ] Am I in the right working directory?
- [ ] Do I have a clear verification criterion ("the test that should pass when this is done")?

## When to `/clear`, `/compact`, or `/rewind`

| Symptom | Use | Why |
|---|---|---|
| Starting a new unrelated task | `/clear` | Fresh context; previous task's noise gone |
| Long session, context filling | `/compact` | Summarizes tool outputs to free space |
| Claude made a bad decision recently | `/rewind` | Backs out to a checkpoint without losing earlier work |
| Two failed attempts at the same fix | `/clear` + retry with better prompt | Two failures = signal your prompt is the problem, not Claude's effort |
