_D2.1 designed a tool's happy path; this chapter designs its failure path. When a call goes wrong, the architect decides three things: which channel the failure travels down, what the failure text says, and whether the schema could have prevented it at all. Get them right and a failed call becomes a recoverable signal the model acts on; get them wrong and the agent stalls, guesses, or apologizes. The exam tests the channel split and the difference between steering a retry and preventing the error._

## `is_error: true` is the canonical failure signal

When a tool fails, the host still returns a `tool_result` — but flags it: "`is_error: true` is the canonical signal that a tool call failed. Claude incorporates the error into its next-turn reasoning and may retry."<Tag kind="official" /><Citation src="anthropic-handle-tool-calls" /> The flag is what turns a failure into a *message to the model* rather than a dead end — a result whose content reads `ConnectionError: the weather service API is not available (HTTP 500)` with `is_error: true` lets the next turn reason about what to do.<Tag kind="official" /><Citation src="anthropic-handle-tool-calls" />

The design principle is two lines: set `is_error: true` on the `tool_result` block, and make the `content` text actionable.<Tag kind="official" /><Citation src="writing-tools-for-agents" />

<KeyIdea>
An error is a message to the model, not a log line for you. Its only job is to let the next turn recover — so it must be both *flagged* (so the model knows the call failed) and *legible* (so the model knows what to do about it).
</KeyIdea>

## Write instructive error messages

The flag says *that* it failed; the content must say *what to do next*. The documented principle is explicit: "Write instructive error messages. Instead of generic errors like 'failed', include what went wrong and what Claude should try next, e.g., 'Rate limit exceeded. Retry after 60 seconds.' This gives Claude the context it needs to recover or adapt without guessing."<Tag kind="official" /><Citation src="anthropic-handle-tool-calls" />

<Pitfall title="The 'failed' anti-pattern">
A `tool_result` of `"failed"` or a raw stack trace gives the model nothing to act on — it guesses or gives up. Every error string should name what went wrong *and* what to try next: not `"Error: 429"` but `"Rate limit exceeded. Retry after 60 seconds."` That actionable form is the difference between a recovered task and a stalled one.
</Pitfall>

## Execution errors vs protocol errors: pick the channel

A failure can travel down one of two channels, and the choice is normative, not stylistic. The MCP specification draws the line: `isError: true` *inside a successful result* is for **execution errors the model should self-correct on** — input validation, API failures, business-logic errors — while a **JSON-RPC error response** is for **protocol errors the model cannot fix**, such as an unknown tool or a malformed request.<Tag kind="official" /><Citation src="mcp-spec-tools" />

The most-tested trap lives on this line: a validation failure belongs in the `isError` channel, not in a JSON-RPC `-32602`. The 2025-11-25 spec (per SEP-1303) is explicit that input-validation errors return as `isError: true` content — for example, `Invalid departure date: must be in the future. Current date is 08/08/2025.` — so the model can correct and retry.<Tag kind="official" /><Citation src="mcp-spec-tools" />

<ConceptBox title="Two channels, two audiences">
The `isError` content is addressed to the **model** — it reads the text and self-corrects. The JSON-RPC error is addressed to the **client/host** — the model cannot usefully recover from it, so routing a recoverable failure there silently denies the model its chance to retry. Route by *who can act on the error*, not by how severe it feels.
</ConceptBox>

## Retryability is documented behavior, not a parameter

The model already retries failed calls on its own: "If a tool request is invalid or missing parameters, Claude will retry 2-3 times with corrections before apologizing to the user."<Tag kind="official" /><Citation src="anthropic-handle-tool-calls" /> That loop is *why* the channel choice and the content quality matter so much — a failure returned as legible `isError` content feeds each retry something to correct against, whereas a protocol error the model cannot read gives it nothing to adjust and burns the budget toward an apology.

<MarginNote variant="note" label="Note">
The 2–3-times retry is documented *default behavior*, not an API knob — there is no retry-count parameter. The only lever over the retry is the quality of the error content the model reads on each attempt. Good error text is how you steer a retry you cannot configure.
</MarginNote>

## Prevent the error: `strict: true`

The cheapest error to handle is the one that never happens. For the largest class of failures — malformed inputs — there is a prevention switch: "To eliminate invalid tool calls entirely, use strict tool use with `strict: true` on your tool definitions. This guarantees that tool inputs will always match your schema exactly, preventing missing parameters and type mismatches."<Tag kind="official" /><Citation src="anthropic-handle-tool-calls" /><Citation src="anthropic-define-tools" /> With `strict: true` the schema-violation error class disappears before it can reach your handler.

<Pitfall title="Handling what you could have prevented">
Writing elaborate validation-error content for a parameter that `strict: true` would have made impossible is wasted effort. Reserve `isError` content for the failures you *cannot* prevent — runtime API errors, business-logic violations, and semantic constraints a JSON Schema can't express (like "the date must be in the future"). Prevent the structural errors; make the rest legible.
</Pitfall>

## Exam essentials

- **`is_error: true` is the canonical failure signal** — it turns a failed `tool_result` into a message the model reasons over and may retry against. Flag it *and* make the content actionable.
- **Write instructive error text** — "Rate limit exceeded. Retry after 60 seconds." beats `"failed"`; name what went wrong and what to try next.
- **Two channels, two audiences** — `isError: true` content = execution errors the model self-corrects (validation, API, business logic); a JSON-RPC error = protocol errors it cannot fix (unknown tool, malformed request). Validation errors go in the `isError` channel (SEP-1303), never JSON-RPC `-32602`.
- **Retry is default behavior, not a parameter** — Claude retries 2–3 times with corrections; there is no retry-count knob, so error-content quality is your only lever over it.
- **Prevent with `strict: true`** — eliminates schema-violation errors entirely; reserve `isError` content for runtime, business-logic, and semantic failures you cannot prevent.
